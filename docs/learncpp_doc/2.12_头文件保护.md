# 2.12 — 头文件保护（Header guards）

---

## “重复定义”的问题

在 [2.7 — 前向声明与定义](#) 中我们提到：**一个变量或函数标识符只能有一个定义（一次定义原则 ODR）**。因此，**在同一个文件里定义两次同名变量**会导致编译错误：

```cpp
int main()
{
    int x; // 定义变量 x
    int x; // 编译错误：重复定义

    return 0;
}
```

同样，**在同一个文件里定义两次同名函数**也会编译错误：

```cpp
#include <iostream>

int foo() // 定义函数 foo
{
    return 5;
}

int foo() // 编译错误：重复定义
{
    return 5;
}

int main()
{
    std::cout << foo();
    return 0;
}
```

上面这些例子只要**删掉重复定义**就行。但**一旦引入头文件**，就很容易出现：**头文件里的某个定义被“多次”包含进同一个源文件**。这在**头文件 A `#include` 了头文件 B**（很常见）时尤其容易发生。

> **作者笔记**
>
> 接下来的例子里，我们会**在头文件中定义函数**。**通常你不应该这么做。** 这里这样做只是为了**用已经学过的东西**最有效地演示概念。

看下面这个学院派例子：

**`square.h`**：

```cpp
int getSquareSides()
{
    return 4;
}
```

**`wave.h`**：

```cpp
#include "square.h"
```

**`main.cpp`**：

```cpp
#include "square.h"
#include "wave.h"

int main()
{
    return 0;
}
```

这段看似人畜无害的程序**编不过**！过程是这样的：

- `main.cpp` 先 `#include "square.h"`，把 `getSquareSides` 的**定义**复制进 `main.cpp`；
- `main.cpp` 再 `#include "wave.h"`，而 `wave.h` 自己又 `#include "square.h"`，把 `square.h` 的内容（含 `getSquareSides` 的定义）复制进 `wave.h`，再随之复制进 `main.cpp`。

所以所有 `#include` 展开后，`main.cpp` 变成了：

```cpp
int getSquareSides()  // 来自 square.h
{
    return 4;
}

int getSquareSides()  // 来自 wave.h（其内部又来自 square.h）
{
    return 4;
}

int main()
{
    return 0;
}
```

**重复定义 → 编译错误**。每个文件**单看都没问题**，但因为 `main.cpp` 最终把 `square.h` 的内容**塞了两次**，就出事了。如果 `wave.h` 需要用到 `getSquareSides()`，而 `main.cpp` 又需要用到 `wave.h` 和 `square.h`，该怎么解决？

---

## 头文件保护（Header guards）

好消息是，我们可以用一种叫 **头文件保护（header guard / include guard）** 的机制来避免这种问题。头文件保护是**条件编译指令**，一般写成这样：

```cpp
#ifndef SOME_UNIQUE_NAME_HERE
#define SOME_UNIQUE_NAME_HERE

// 你的声明（以及某些类型的定义）放在这里

#endif
```

当这个头文件被 `#include` 时，预处理器会检查 `SOME_UNIQUE_NAME_HERE` 是否**已经在当前翻译单元中被 `#define` 过**：

- 如果**第一次**包含这个头，`SOME_UNIQUE_NAME_HERE` 还没被定义，就会 `#define` 它，并**包含该头文件里的内容**；
- 如果**再次**被 `#include` 进同一个文件，`SOME_UNIQUE_NAME_HERE` 早就被定义了，因此**头文件里的内容会被 `#ifndef` 屏蔽掉**。

**你的每个头文件都应该带头文件保护。** `SOME_UNIQUE_NAME_HERE` 可以取任何名字，但**惯例上**用**头文件的完整名字全大写、下划线代替空格或标点**。例如 `square.h` 的头文件保护写作 `SQUARE_H`：

**`square.h`**：

```cpp
#ifndef SQUARE_H
#define SQUARE_H

int getSquareSides()
{
    return 4;
}

#endif
```

标准库头文件本身也使用头文件保护。比如 Visual Studio 的 `iostream` 大致长这样：

```cpp
#ifndef _IOSTREAM_
#define _IOSTREAM_

// 内容

#endif
```

> **进阶阅读**
>
> 在大型项目中，可能出现**两个不同目录**下**同名头文件**的情况（例如 `directoryA\config.h` 和 `directoryB\config.h`）。如果它们的头文件保护宏都只按“文件名”命名（例如都用 `CONFIG_H`），就会**撞名**——一旦某个源文件（直接或间接地）同时包含了这两个 `config.h`，**第二个被包含的那份内容会被误屏蔽**，可能引发编译错误。
>
> 因此很多开发者建议用**更独特、更复杂**的宏名，例如：
> `PROJECT_PATH_FILE_H`、`FILE_LARGE-RANDOM-NUMBER_H`、`FILE_CREATION-DATE_H` 等。

## 给之前的例子加上头文件保护

回到 `square.h` 的例子，给它加上头文件保护；同时为了规范，`wave.h` 也加上保护。

**`square.h`**：

```cpp
#ifndef SQUARE_H
#define SQUARE_H

int getSquareSides()
{
    return 4;
}

#endif
```

**`wave.h`**：

```cpp
#ifndef WAVE_H
#define WAVE_H

#include "square.h"

#endif
```

**`main.cpp`**：

```cpp
#include "square.h"
#include "wave.h"

int main()
{
    return 0;
}
```

预处理器展开所有 `#include` 之后，`main.cpp` 大致等价于：

```cpp
// 来自 main.cpp 的第一次 #include "square.h"
#ifndef SQUARE_H
#define SQUARE_H

int getSquareSides()
{
    return 4;
}

#endif // SQUARE_H

#ifndef WAVE_H   // 来自 main.cpp 的 #include "wave.h"
#define WAVE_H

#ifndef SQUARE_H // 来自 wave.h 内部的 #include "square.h"，但 SQUARE_H 已经被定义过了
#define SQUARE_H // 所以下面这一整段都不会再被包含

int getSquareSides()
{
    return 4;
}

#endif // SQUARE_H
#endif // WAVE_H

int main()
{
    return 0;
}
```

过程解读：

1. 预处理器先看到 `#ifndef SQUARE_H`。`SQUARE_H` 还没定义，所以中间那段会参与编译：这段代码**定义了 `SQUARE_H`**，并给出 `getSquareSides` 的定义；
2. 后面又出现一次 `#ifndef SQUARE_H`。这次 `SQUARE_H` 已经被定义（上一步定义的），因此中间的内容**被排除在编译之外**。

**头文件保护的核心作用**：**第一次遇到时**保护宏还没定义，被保护的内容会被引入；**之后再遇到**保护宏已经存在，重复内容就会被跳过。

## 头文件保护**不能**阻止头文件被“多次包含进不同的源文件”

要注意：**头文件保护的目标是防止一个源文件里出现同一个头文件的多份拷贝**；它**并不**阻止一个头文件被**分别**包含进**多个不同的源文件**（各一次）。这一点如果没注意，也会踩坑：

**`square.h`**：

```cpp
#ifndef SQUARE_H
#define SQUARE_H

int getSquareSides()
{
    return 4;
}

int getSquarePerimeter(int sideLength); // 前向声明

#endif
```

**`square.cpp`**：

```cpp
#include "square.h" // 这里被包含一次

int getSquarePerimeter(int sideLength)
{
    return sideLength * getSquareSides();
}
```

**`main.cpp`**：

```cpp
#include "square.h" // 这里也被包含一次
#include <iostream>

int main()
{
    std::cout << "a square has " << getSquareSides() << " sides\n";
    std::cout << "a square of length 5 has perimeter length " << getSquarePerimeter(5) << '\n';

    return 0;
}
```

`square.h` 被 `main.cpp` 和 `square.cpp` **各自**都包含了一次——**每个源文件都会拿到一份 `square.h` 的内容**。

再细看一下：当 `square.h` 被 `square.cpp` 包含时，`SQUARE_H` 会在 `square.cpp` 中一直保持“被定义”到该文件末尾（这保证了它不会在同一个源文件里被重复包含）。**但 `square.cpp` 编译结束后，这个宏也就“作废”了**——当预处理器开始处理 `main.cpp` 时，**`SQUARE_H` 又变成“未定义”**。

结果就是：`square.cpp` 和 `main.cpp` **各得到一份 `getSquareSides` 的定义**。**编译能过，链接会报错**：`getSquareSides` 有**多个定义**。

**解决办法很简单**：把函数定义放到 `.cpp` 里，头文件里只留**前向声明**：

**`square.h`**：

```cpp
#ifndef SQUARE_H
#define SQUARE_H

int getSquareSides();                    // 前向声明
int getSquarePerimeter(int sideLength);  // 前向声明

#endif
```

**`square.cpp`**：

```cpp
#include "square.h"

int getSquareSides() // 真正的定义
{
    return 4;
}

int getSquarePerimeter(int sideLength)
{
    return sideLength * getSquareSides();
}
```

**`main.cpp`**：

```cpp
#include "square.h"
#include <iostream>

int main()
{
    std::cout << "a square has " << getSquareSides() << " sides\n";
    std::cout << "a square of length 5 has perimeter length " << getSquarePerimeter(5) << '\n';

    return 0;
}
```

这样，`getSquareSides` 在整个程序里就**只有一个定义**（在 `square.cpp` 里），链接器就满意了。`main.cpp` 之所以还能调用它，是因为它 `#include` 了 `square.h`，里面有前向声明——链接器会**把 `main.cpp` 里对 `getSquareSides` 的调用**和 **`square.cpp` 里 `getSquareSides` 的定义**连接起来。

## 既然头文件里本来就不该写定义，那还需要头文件保护吗？

我们之前一直强调**不要**把函数定义写进头文件。那你可能会问：**头文件保护要防的事，本来就不该做，还要它干嘛？**

后面会看到很多**必须**在头文件里放**非函数定义**的情况。例如，C++ 允许你**自定义类型**，这些自定义类型**通常就定义在头文件中**，以便传播到所有要用它们的源文件里。**没有头文件保护的话**，一个源文件很容易得到**多份（内容相同的）类型定义**，编译器会因此报错。

所以尽管现阶段“非要它不可”的场景还不多，我们**现在就养成好习惯**，以后就不用回头改坏习惯。

## `#pragma once`

现代编译器还支持一种**更简单的头文件保护写法**——预处理指令 `#pragma`：

```cpp
#pragma once

// 你的代码
```

`#pragma once` 的作用**和头文件保护相同**：防止一个头文件被多次包含。

- 传统写法（`#ifndef / #define / #endif`）由**开发者负责**保护；
- `#pragma once` 则是**请编译器**来做保护，具体怎么实现是**实现细节**。

> **进阶阅读**
>
> `#pragma once` 有一种已知的失效情形：**同一份头文件被复制到文件系统的多个位置**。如果两份拷贝都被包含进来，**传统头文件保护**会成功去重（宏名相同）；而 **`#pragma once` 可能失效**（编译器未必意识到这两份文件“内容其实完全相同”）。

对绝大多数项目来说，`#pragma once` 都工作得很好，也**更简单、更少出错**，很多 IDE 在生成新头文件时会**自动加上 `#pragma once`**。

> **警告**
>
> `#pragma` 指令本来就是**留给各编译器厂商**自行使用的机制——**支持哪些 pragma、含义是什么，完全由实现决定**。**除了 `#pragma once` 之外**，别指望在一个编译器上能用的 pragma 会在别的编译器上被支持。

由于 `#pragma once` 并不在 C++ 标准里，**理论上**存在某些编译器不实现它的可能。也正因如此，一些团队（例如 Google）**推荐使用传统的头文件保护**。本教程也**默认使用传统的头文件保护**——这是最主流、最规范的写法。不过 `#pragma once` 现在几乎已经被普遍支持，若你选用 `#pragma once`，在现代 C++ 中**同样是被广泛接受的**。

---

## 小结

- **头文件保护**的目的是：**确保同一个头文件的内容不会被多次拷贝进同一个源文件**，从而避免重复定义。
- **重复的声明**是**允许**的，但即便你的头文件里**全是声明**、没有任何定义，**加上头文件保护仍然是最佳实践**。
- 头文件保护**不会**阻止一个头文件被“各自一次”地拷贝到**不同的源文件**中。这其实是一件好事——因为不同源文件本来就常常需要引用同一个头文件里的内容。
