# 2.7 — 前向声明与定义

---

看下面这段看起来人畜无害的示例程序：

```cpp
#include <iostream>

int main()
{
    std::cout << "The sum of 3 and 4 is: " << add(3, 4) << '\n';
    return 0;
}

int add(int x, int y)
{
    return x + y;
}
```

你可能会期待它输出：

```
The sum of 3 and 4 is: 7
```

但事实上，**它根本编译不过！** 在 Visual Studio 上编译会得到类似这样的错误：

```
add.cpp(5) : error C3861: 'add': identifier not found
```

编译不过的原因是：**编译器是按顺序编译源代码的**。当编译器读到 `main` 里第 5 行的 `add()` 时，它**根本不知道 `add` 是什么**——因为我们直到第 9 行才定义 `add`！于是就报了 “identifier not found”（找不到该标识符）。

老版本的 VS 还可能额外报一条：

```
add.cpp(9) : error C2365: 'add'; : redefinition; previous definition was 'formerly unknown identifier'
```

这条其实**有点误导**——`add` 从来就没被定义过。但从中可以学到一个经验：**一个错误往往会“连带”产生一堆冗余或相关的报错/警告**。有时你会分不清后面的错是**第一个错的连锁反应**，还是**独立的问题**。

> **最佳实践**
>
> **修编译错误/警告时，先处理列表中的第一个，然后重新编译。**

要解决这个问题，我们要让编译器**知道 `add` 是什么**。有两种常见做法。

---

## 方法一：调整函数定义的顺序

把 `add` 挪到 `main` 前面：

```cpp
#include <iostream>

int add(int x, int y)
{
    return x + y;
}

int main()
{
    std::cout << "The sum of 3 and 4 is: " << add(3, 4) << '\n';
    return 0;
}
```

这样当 `main` 调用 `add` 时，编译器就已经“认识” `add` 了。程序小的时候没问题，但**在大型项目里**，你要搞清楚“谁调用了谁、按什么顺序放”是**非常烦人**的。

而且这种方式**并不总是可行**。比如程序里有函数 `A` 和 `B`：**如果 `A` 调用 `B`、`B` 又调用 `A`**，那**无论谁放前面**都不行——先放 `A`，编译器会抱怨不认识 `B`；先放 `B`，又不认识 `A`。

---

## 方法二：使用前向声明（forward declaration）

我们还可以用**前向声明**来解决。

> **前向声明允许我们在正式定义某个标识符之前，就告诉编译器它的存在。**

对函数来说，这意味着我们可以**先告诉编译器“这里有这么一个函数”**，之后再给它写函数体。这样，当编译器碰到对该函数的调用时，就知道这是一次函数调用，可以**检查调用是否合法**，即便它此时还不知道函数具体定义在哪、怎么实现。

**函数的前向声明**通过一条**函数声明语句**（也叫**函数原型 function prototype**）来完成。它包含函数的：**返回类型、函数名、形参类型**，末尾加一个分号；**形参名可写可不写**；**函数体不写**。

例如 `add` 函数的声明：

```cpp
int add(int x, int y); // 函数声明：包含返回类型、函数名、形参类型和分号。没有函数体！
```

用它作为 `add` 的前向声明，就能修好之前那段程序：

```cpp
#include <iostream>

int add(int x, int y); // add() 的前向声明（通过函数声明来做）

int main()
{
    std::cout << "The sum of 3 and 4 is: " << add(3, 4) << '\n'; // 有了前向声明，这里就能编译通过
    return 0;
}

int add(int x, int y) // 尽管 add() 的函数体到这里才定义
{
    return x + y;
}
```

这样编译器在遇到 `main` 里的 `add` 调用时，就已经知道：`add` 是一个“接受两个 int、返回一个 int 的函数”，从而不会报错。

值得一提的是，**函数声明中并不需要写形参名字**（形参名不属于函数声明的一部分）。例如你也可以这样声明：

```cpp
int add(int, int); // 合法的函数声明
```

但我们**更推荐带上形参名字**（并与函数实际定义一致）。这样，只看声明就能明白参数是什么含义。比如你看到 `void doSomething(int, int, int)`，你可能自认为“记得每个参数是干嘛的”，但**很可能记错**。

另外，**很多自动生成文档的工具**是从头文件里的声明来生成文档的（我们会在 2.11 — 头文件 中讲到）。

> **最佳实践**
>
> **函数声明中保留形参名字。**

> **小贴士**
>
> 想快速写函数声明？直接**复制函数头，加上一个分号**就行。

---

## 为什么要用前向声明？

你可能会问：既然重排函数顺序也能解决，为什么还要前向声明？

- **最常见的用途**：告诉编译器**“某个函数存在，它定义在别的源文件里”**。这种场景**根本没法通过“调整顺序”解决**，因为调用方和被调用方在**完全不同的文件**里！我们会在下一节 [2.8 — 多源文件的程序](#) 中详细讲。
- **让函数定义顺序自由**：可以按“把相关函数聚在一起”“阅读顺序更自然”等标准组织代码，而不用被“谁在前谁在后”束缚。
- **解决循环依赖**：偶尔会遇到**两个函数互相调用**的情形，重排顺序无法同时满足，**前向声明可以打破这种循环依赖**。

---

## 忘了写函数体会怎样？

新手常问：如果我**前向声明了某个函数，却没定义它**，会发生什么？

答案是 **“看情况”**：

- 如果**声明了但从没调用**，程序**能正常编译、运行**；
- 如果**声明了并调用了**，但整个程序里**从没有它的定义**，那么**编译能过**，但**链接器会报错**，因为找不到这个函数的实现。

例子：

```cpp
#include <iostream>

int add(int x, int y); // add() 的前向声明

int main()
{
    std::cout << "The sum of 3 and 4 is: " << add(3, 4) << '\n';
    return 0;
}

// 注意：整段代码里 add 都没有定义
```

在 VS 中会得到类似：

```
Compiling...
add.cpp
Linking...
add.obj : error LNK2001: unresolved external symbol "int __cdecl add(int,int)" (?add@@YAHHH@Z)
add.exe : fatal error LNK1120: 1 unresolved externals
```

可以看到：**编译过了，但链接阶段失败**，原因就是 `int add(int, int)` **从未被定义**。

## 其它类型的前向声明

前向声明**最常用于函数**，但在 C++ 中也可以用于**变量、类型**等其它标识符。它们的前向声明语法不一样，之后的课程再讲。

---

## 声明 vs 定义

你会经常听到 “declaration（声明）” 和 “definition（定义）” 这两个词，甚至常被混着用。它们到底有什么区别？现在你已经具备了理解它们的基础。

**声明**：告诉编译器**某个标识符的存在**以及它的**类型信息**。例如：

```cpp
int add(int x, int y); // 告诉编译器：有一个叫 add 的函数，接收两个 int，返回一个 int。没有函数体！
int x;                 // 告诉编译器：有一个叫 x 的 int 变量
```

**定义**：一种**同时也把标识符“实现出来（对函数/类型）”或“实例化出来（对变量）”** 的声明。例如：

```cpp
// 有函数体，所以这是 add() 的一个实现（定义）
int add(int x, int y)
{
    int z{ x + y };   // 实例化了变量 z

    return z;
}

int x; // 实例化了变量 x
```

在 C++ 中，**所有定义都是声明**。因此 `int x;` **既是声明，也是定义**。

反过来，**并不是所有声明都是定义**。**不做实现/实例化的声明**称为 **纯声明（pure declaration）**。纯声明包括：函数、变量和类型的**前向声明**。

> **命名约定**
>
> 日常表达中，人们常把 **“declaration”** 特指 **“pure declaration（纯声明）”**，把 **“definition”** 特指 **“既是定义也是声明”**。因此 `int x;` 我们通常直接叫它“定义”，尽管它同时也是声明。

编译器碰到一个标识符时，会检查它的使用是否合法（是否在作用域内、是否语法合规等等）。**大多数情况下，一个声明就足以让编译器验证标识符的用法。** 比如遇到函数调用 `add(5, 6)`，只要编译器之前**见过 `add(int, int)` 的声明**，就能验证 `add` 是一个接收两个 int 的函数，即使**它的实现还没见过**（可能在另一个文件里）。

不过，**有些场景编译器必须能看到完整的定义**才能使用某个标识符（比如**模板定义**和**类型定义**，以后会讲）。

小结表：

| 术语                           | 技术含义                                    | 示例                                                         |
| ------------------------------ | ------------------------------------------- | ------------------------------------------------------------ |
| **声明（Declaration）**        | 告诉编译器某个标识符存在及其类型信息        | `void foo();` // 函数前向声明（无函数体）<br>`void goo() {};` // 函数定义（有函数体）<br>`int x;` // 变量定义 |
| **定义（Definition）**         | 实现函数 / 实例化变量。**定义同时也是声明** | `void foo() { }` // 函数定义<br>`int x;` // 变量定义         |
| **纯声明（Pure declaration）** | 不是定义的声明                              | `void foo();` // 函数前向声明                                |
| **初始化（Initialization）**   | 给已定义的对象提供初始值                    | `int x { 2 };` // 把 x 初始化为 2                            |

（表中示例列使用了日常口语的说法：“declaration”常指“纯声明”，“definition”指“既是定义也是声明”。）

## 一次定义原则（One Definition Rule，ODR）

**一次定义原则（ODR）** 是 C++ 中一条大名鼎鼎的规则，包含三部分：

1. **在一个文件内**：每个函数、变量、类型或模板在**给定作用域**中**只能有一个定义**。发生在**不同作用域**的定义（例如**不同函数里各自的局部变量**、**不同命名空间里的函数**）**不违反**这条。
2. **在一个程序内**：每个函数或变量在**给定作用域**中**只能有一个定义**。（程序可以由多个源文件组成，见下一节。）**对链接器不可见的函数和变量**不受此约束（详见 7.6 — 内部链接）。
3. **类型、模板、内联函数、内联变量**允许在**不同文件中出现重复定义**，前提是**每个定义都完全相同**。这些概念大多还没讲，先不用担心，用得着的时候我们会再提。

> **相关内容**
>
> 关于 ODR 第 3 条的“豁免”，我们在以下课程中会更详细讨论：
>
> - **类型**：13.1 — 程序自定义（用户自定义）类型简介
> - **函数模板**：11.6 — 函数模板；11.7 — 函数模板实例化
> - **内联函数与内联变量**：7.9 — 内联函数与内联变量

违反 ODR 的后果：

- **违反第 1 条** → 编译器报 **重定义错误**；
- **违反第 2 条** → 链接器报 **重定义错误**；
- **违反第 3 条** → **未定义行为**。

下面这段代码违反了第 1 条：

```cpp
int add(int x, int y)
{
    return x + y;
}

int add(int x, int y) // 违反 ODR：add(int, int) 已经定义过了
{
    return x + y;
}

int main()
{
    int x{};
    int x{ 5 }; // 违反 ODR：x 在 main 的作用域内已定义过
}
```

`add(int, int)` 在全局作用域被定义了两次，`main()` 里的 `int x` 也被定义了两次。Visual Studio 会给出类似这样的错误：

```
project3.cpp(9): error C2084: function 'int add(int,int)' already has a body
project3.cpp(3): note: see previous definition of 'add'
project3.cpp(16): error C2086: 'int x': redefinition
project3.cpp(15): note: see declaration of 'x'
```

不过，`main()` 里的局部变量叫 `int x`，而 `add()` 的形参也叫 `int x`，**这并不违反 ODR 第 1 条**——它们分别位于**各自函数的作用域**里，是**两个不同对象的定义**，而不是“对同一个对象的重复定义”。

> **进阶阅读**
>
> **同名但形参列表不同的函数**会被视为**不同的函数**，因此这类定义**不违反 ODR**。我们会在 11.1 — 函数重载简介 中详细讨论。
