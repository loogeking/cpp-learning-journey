# 2.5 — 局部作用域简介

---

## 局部变量（Local variables）

**定义在函数体内的变量称为局部变量（local variable）**（与之相对的是**全局变量（global variable）**，我们会在后面的章节讲）：

```cpp
int add(int x, int y)
{
    int z{ x + y }; // z 是局部变量

    return z;
}
```

**函数的形参也通常被视为局部变量**，本教程中也会这么归类：

```cpp
int add(int x, int y) // 形参 x 和 y 也是局部变量
{
    int z{ x + y };

    return z;
}
```

本节我们会更详细地看看局部变量的一些性质。

---

## 局部变量的生命周期（Lifetime）

在 [1.3 — 对象与变量简介](1.3_对象和变量简介.md) 中我们提到过：`int x;` 这样的定义语句被执行时，变量会被**实例化**。**形参在函数进入时被创建并初始化，函数体内的变量则在其定义处被创建并初始化**。

例如：

```cpp
int add(int x, int y) // x 和 y 在这里被创建并初始化
{
    int z{ x + y };   // z 在这里被创建并初始化

    return z;
}
```

接下来的自然问题是：**一个被实例化的变量什么时候会被销毁？** —— **局部变量会在其所在的一对大括号结束时被销毁**（对于函数形参来说是函数结束时），**销毁顺序与创建顺序相反**：

```cpp
int add(int x, int y)
{
    int z{ x + y };

    return z;
} // z、y、x 在这里被销毁
```

就像“人的生命周期”是从出生到死亡一样，**对象的生命周期**就是**从它被创建到被销毁的这段时间**。要注意的是：**变量的创建与销毁发生在程序运行时（runtime）**，而不是编译期。因此，**生命周期是运行时属性**。

> **进阶阅读**
>
> 上面这些关于“创建、初始化、销毁时机”的规则是 **保证**：即对象**必须**在**不晚于其定义处**被创建并初始化，**不早于其所在大括号结束**（或对于形参：函数结束）时被销毁。
>
> 实际上，C++ 标准给了编译器**很大的自由度**来决定局部变量的具体创建/销毁时机。出于优化目的，对象可能被**提前创建**或**延后销毁**。最常见的情况是：**函数进入时创建所有局部变量，函数退出时按相反顺序销毁**。等我们讨论到**调用栈**时会更详细讲。

下面是一个稍复杂点的例子，展示变量 `x` 的生命周期：

```cpp
#include <iostream>

void doSomething()
{
    std::cout << "Hello!\n";
}

int main()
{
    int x{ 0 };    // x 的生命周期从这里开始

    doSomething(); // 在这次函数调用期间，x 仍然“活着”

    return 0;
} // x 的生命周期在这里结束
```

上面这段程序中，`x` 的生命周期从定义处一直持续到 `main` 结束，**包括 `doSomething` 执行期间**。

---

## 对象被销毁时会发生什么？

大多数情况下：**什么都不会发生**，被销毁的对象只是**变得不再有效**。

> **进阶阅读**
>
> 如果对象是**类类型对象**，那么在销毁前，会调用一个叫做**析构函数（destructor）** 的特殊函数。很多时候析构函数什么都不做，也就没什么额外开销。我们会在 15.4 — 析构函数简介 中介绍析构函数。

**在对象被销毁之后再使用它，会导致未定义行为。**

在销毁发生之后的某个时刻，这个对象所占用的**内存会被释放**（重新变得可用）。

## 局部作用域（Local scope / Block scope）

**一个标识符的“作用域（scope）”决定了它在源代码的哪些位置可以被看到和使用。** 能看到、能用时，我们说它 **“在作用域内（in scope）”**；看不到时就 **“在作用域外（out of scope）”**，此时不能使用。**作用域是编译期属性**，尝试在作用域外使用标识符会导致**编译错误**。

**局部变量的标识符具有“局部作用域”**（技术上称为 **“块作用域（block scope）”**）。具有局部作用域的标识符从**定义处**开始可用，直到**包含它的最内层那一对大括号结束**为止（对形参而言：直到函数结束）。这确保了：局部变量**不会**在**定义之前**被使用（即便编译器可能提前创建了它），也**不会**在**销毁之后**被使用。**在一个函数里定义的局部变量，在它调用的其它函数里也是不可见的。**

下面这个例子展示了 `x` 的作用域：

```cpp
#include <iostream>

// 在这个函数里，x 处处都不在作用域内
void doSomething()
{
    std::cout << "Hello!\n";
}

int main()
{
    // 这里还不能使用 x，因为它还没进入作用域

    int x{ 0 }; // x 从这里进入作用域，之后可以在本函数中使用

    doSomething();

    return 0;
} // x 在这里离开作用域，无法再使用
```

`x` 在定义处进入作用域，在**最内层包含它的一对大括号**结束时离开作用域——也就是 `main()` 的右大括号处。注意：**在 `doSomething` 里任何位置，`x` 都不在作用域内**。“`main` 调用了 `doSomething`”这一事实在这里**并不影响作用域**。

---

## “out of scope” 与 “going out of scope” 的区别

新手常把这两个术语搞混。

- **out of scope（在作用域之外）** 指的是**标识符**在代码里“看不到、访问不到”的地方。上例中，标识符 `x` 从定义处到 `main` 结束这一段之外的地方，都属于 out of scope。
- **going out of scope（离开作用域）** 通常是用来描述**对象**的：**一个对象在其所在作用域（右大括号处）结束时“离开作用域”**。上例中，名为 `x` 的对象在 `main` 结束时离开作用域。

**局部变量的生命周期在它离开作用域时结束**，因此局部变量此时会被销毁。

不过要注意，**并不是所有类型的变量在离开作用域时都会立刻被销毁**（后面章节会看到例子）。

## 另一个例子

再来一个稍复杂的例子。记住：**生命周期是运行时属性，作用域是编译期属性**，虽然我们会在同一段程序中同时谈到它们，但它们在**不同阶段**被“执行”。

```cpp
#include <iostream>

int add(int x, int y) // x 和 y 在这里被创建、并进入作用域
{
    // x 和 y 只在 add() 内部可用
    return x + y;
} // y 和 x 在这里离开作用域并被销毁

int main()
{
    int a{ 5 }; // a 在这里被创建、初始化并进入作用域
    int b{ 6 }; // b 在这里被创建、初始化并进入作用域

    // a 和 b 只在 main() 内部可用

    std::cout << add(a, b) << '\n'; // 调用 add(5, 6)，此时 x=5, y=6

    return 0;
} // b 和 a 在这里离开作用域并被销毁
```

形参 `x` 和 `y` 在 `add` 被调用时创建，只在 `add` 内可见/可用，在 `add` 结束时销毁。变量 `a` 和 `b` 在 `main` 内创建，只在 `main` 内可见/可用，在 `main` 结束时销毁。

为了帮助你把这些串起来，我们来**按顺序**跟着这段程序走一遍：

1. 执行从 `main` 顶部开始；
2. `main` 的局部变量 `a` 被创建，赋值为 `5`；
3. `main` 的局部变量 `b` 被创建，赋值为 `6`；
4. 使用实参 `5` 和 `6` 调用 `add`；
5. `add` 的形参 `x` 和 `y` 被创建，分别用 `5` 和 `6` 初始化；
6. 求值 `x + y` 得到 `11`；
7. `add` 把 `11` 拷贝返回给调用方 `main`；
8. `add` 的形参 `y` 和 `x` 被销毁；
9. `main` 把 `11` 打印到控制台；
10. `main` 向操作系统返回 `0`；
11. `main` 的局部变量 `b` 和 `a` 被销毁。

完成。

注意：如果 `add` 被调用**两次**，形参 `x` 和 `y` 就会**被创建和销毁两次**——每次调用各一次。在一个函数众多、调用频繁的程序中，变量会被**频繁地创建和销毁**。

## 各自独立（Functional separation）

上面例子里，`a`、`b` 和 `x`、`y` 的区别一眼就看出来了。

再看下面这个类似程序：

```cpp
#include <iostream>

int add(int x, int y) // add 的 x、y 在这里创建、进入作用域
{
    // add 的 x、y 只在本函数中可见/可用
    return x + y;
} // add 的 y、x 在这里离开作用域并被销毁

int main()
{
    int x{ 5 }; // main 的 x 在这里创建、初始化、进入作用域
    int y{ 6 }; // main 的 y 在这里创建、初始化、进入作用域

    // main 的 x、y 只在本函数中可用
    std::cout << add(x, y) << '\n'; // 调用 add(5, 6)

    return 0;
} // main 的 y、x 在这里离开作用域并被销毁
```

我们只是把 `main` 里的 `a`、`b` 改叫 `x`、`y`。程序照样能编译运行——尽管 `main` 和 `add` **各自都有一个叫 `x` 和一个叫 `y` 的变量**。为什么可以？

- **首先**，虽然两个函数都定义了 `x` 和 `y`，但**它们是完全不同的变量**。`main` 中的 `x`、`y` 与 `add` 中的 `x`、`y` **毫无关系**，只是恰好重名。
- **其次**，在 `main` 内部，`x` 和 `y` 指的是 `main` 的局部变量，只能在 `main` 里使用；在 `add` 内部，`x` 和 `y` 指的则是 `add` 的形参，只能在 `add` 里使用。

一句话：**`add` 不知道 `main` 里有同名变量，`main` 也不知道 `add` 里有同名形参**。**由于它们的作用域不重叠**，编译器任何时候都能明确“当前的 `x`、`y` 到底指的是谁”。

> **关键洞察**
>
> **函数形参名或函数体内的局部变量名，只在定义它们的函数内可见。** 这意味着不同函数里可以放心使用同名的局部变量，**互不干扰**。这有助于函数彼此**保持独立**。

后面章节会更详细地讨论局部作用域和其它作用域。

## 局部变量该定义在哪里？

**现代 C++ 的最佳实践是：把局部变量定义在“尽可能接近它第一次被使用的位置”**：

```cpp
#include <iostream>

int main()
{
    std::cout << "Enter an integer: ";
    int x{};       // 在这里定义 x
    std::cin >> x; // 紧接着就使用

    std::cout << "Enter another integer: ";
    int y{};       // 在这里定义 y
    std::cin >> y; // 紧接着就使用

    int sum{ x + y }; // 此时才能用“预期的值”来初始化 sum
    std::cout << "The sum is: " << sum << '\n';

    return 0;
}
```

上面每个变量都刚好定义在**首次使用前**。不用太苛求——如果你觉得把提示语（第 5 行）和定义（第 6 行）顺序换一下更顺眼，也完全 OK。

> **最佳实践**
>
> **在合理范围内，把局部变量定义在“尽可能接近首次使用”的地方。**

> **顺便一提**
>
> 由于早期编译器的限制，**C 语言曾要求所有局部变量必须定义在函数开头**。用这种老风格写的等价 C++ 程序会长成这样：
>
> ```cpp
> #include <iostream>
> 
> int main()
> {
>  int x{}, y{}, sum{}; // 这些变量各自是干嘛的？
> 
>  std::cout << "Enter an integer: ";
>  std::cin >> x;
> 
>  std::cout << "Enter another integer: ";
>  std::cin >> y;
> 
>  sum = x + y;
>  std::cout << "The sum is: " << sum << '\n';
> 
>  return 0;
> }
> ```
>
> 这种风格有几个明显缺点：
>
> - 在定义处**看不出这些变量是干什么用的**，要通读整个函数才能弄清楚；
> - 想给出的**初始值可能在函数开头还拿不到**（例如这里 `sum` 就没法在开头用它真正的值初始化，因为还没有 `x`、`y`）；
> - 变量的初始化与它第一次被使用之间可能**隔着很多行**，如果忘了当初赋的什么值，还得滚回函数顶部去看，很打断思路。
>
> 这个限制在 C99 标准中被取消了。

## 什么时候用形参、什么时候用局部变量？

由于**形参和局部变量都能在函数体里使用**，初学者有时不知道该用哪个。**判断标准很简单**：

- **调用方需要以实参形式传入初始值 → 用形参**；
- **否则 → 用局部变量**。

如果**该用局部变量却用了形参**，代码会写成这样（不好）：

```cpp
#include <iostream>

int getValueFromUser(int val) // val 是形参
{
    std::cout << "Enter a value: ";
    std::cin >> val;
    return val;
}

int main()
{
    int x {};
    int num { getValueFromUser(x) }; // main 还要专门定义 x 传进去

    std::cout << "You entered " << num << '\n';

    return 0;
}
```

上面这段里，`val` 是形参，因此 `main` 必须**多定义一个 `x`** 来当实参。但 `x` 的值**从来没被真正使用过**，`val` 的初始值也**没被真正使用过**。**让调用方去定义、去传一个根本用不到的变量，只会白白增加复杂度。**

正确写法：

```cpp
#include <iostream>

int getValueFromUser()
{
    int val {}; // val 是局部变量
    std::cout << "Enter a value: ";
    std::cin >> val;
    return val;
}

int main()
{
    int num { getValueFromUser() }; // main 不用再传任何东西

    std::cout << "You entered " << num << '\n';

    return 0;
}
```

此时 `val` 变成局部变量，`main` 也变得更简单：不用再为 `getValueFromUser` 单独准备一个变量。

> **最佳实践**
>
> 当函数内部需要一个变量时：
>
> - **如果调用方要以实参形式提供它的初始值 → 用形参**；
> - **否则 → 用局部变量**。

## 临时对象（Temporary object）简介

**临时对象（temporary object）**（有时也叫**匿名对象**）是**没有名字的对象**，用来**短暂地保存一个只用一小会儿的值**。临时对象是**编译器在需要时自动生成的**。

临时对象产生的场景有很多，下面是一个常见例子：

```cpp
#include <iostream>

int getValueFromUser()
{
    std::cout << "Enter an integer: ";
    int input{};
    std::cin >> input;

    return input; // 把 input 的值返回给调用方
}

int main()
{
    std::cout << getValueFromUser() << '\n'; // 返回的值到底存到哪去了？

    return 0;
}
```

`getValueFromUser()` 把局部变量 `input` 的值返回给调用方。因为 `input` 会在函数结束时被销毁，所以调用方拿到的是**它的一份拷贝**，这样即使 `input` 被销毁，调用方也还有一个可用的值。

但是——这份**被拷回来的值**存到哪里去了呢？`main` 里我们并**没有定义任何变量**。答案是：**这个返回值被存到了一个临时对象里**，然后再把这个临时对象交给 `std::cout` 打印出来。

> **关键洞察**
>
> **按值返回**其实是把一个**临时对象**（内含返回值的拷贝）返回给调用方。

**临时对象没有任何作用域**（这很合理——作用域是标识符的属性，而临时对象没有标识符）。

**临时对象会在它所处的“完整表达式”末尾被销毁**。也就是说，临时对象总是会**在下一条语句执行之前**就被销毁。

在上面的例子里，用来保存 `getValueFromUser()` 返回值的临时对象，会在 `std::cout << getValueFromUser() << '\n'` 这条完整表达式**执行完之后**被销毁。

**如果临时对象被用来初始化某个变量**，那**初始化会发生在临时对象被销毁之前**。

在现代 C++ 中（尤其是 C++17 之后），编译器有很多手段**避免生成临时对象**——过去必须生成的地方，现在往往可以省略。例如：用一个函数的返回值去**初始化一个变量**时，按理会先生成一个临时对象保存返回值，再用它去初始化那个变量；但在现代 C++ 中，**编译器往往会直接跳过临时对象**，用返回值**直接初始化**那个变量。

类似地，在上面的例子里，`getValueFromUser()` 的返回值紧接着就被输出了，编译器就可以**跳过在 `main` 中生成/销毁临时对象**这一步，**直接用 `getValueFromUser()` 的返回值来初始化 `operator<<` 的形参**。
