# 1.11 — 开发你的第一个程序

---

前面几节课引入了大量术语和概念，几乎在我们写的每一个程序里都会用到。本节我们会**把这些知识整合起来**，一步步写出你的第一个真正意义上的小程序。

---

## 把一个数乘以 2

我们先来写一个程序：**提示用户输入一个整数、等待他们输入、然后告诉他们这个数的 2 倍是多少**。程序运行效果如下（假设我输入的是 4）：

```
Enter an integer: 4
Double that number is: 8
```

那要怎么下手呢？——**一步一步来**。

> **最佳实践**
>
> 新手程序员经常想**一次性把整个程序全写完**，结果编译出一堆错误、心态崩了。更好的策略是：**一次只加一小块，确认它能编译、能正常工作，再往下写下一块。**

我们这里就用这种策略。每一步都请你**亲手在编辑器里输入**（不要复制粘贴），然后编译并运行。

---

### 第 1 步：新建一个控制台项目

先新建一个 Console 项目。

---

### 第 2 步：搭起基本骨架

我们知道每个 C++ 程序都必须有 `main()` 函数。如果你的 IDE 没有自动创建一个，那就自己写一个：

```cpp
int main()
{
    return 0;
}
```

---

### 第 3 步：包含 iostream

我们要向控制台输出文本、也要从键盘读取输入，所以要包含 `<iostream>`，以便使用 `std::cout` 和 `std::cin`：

```cpp
#include <iostream>

int main()
{
    return 0;
}
```

---

### 第 4 步：先提示用户输入

告诉用户我们需要一个整数：

```cpp
#include <iostream>

int main()
{
    std::cout << "Enter an integer: ";

    return 0;
}
```

此时程序运行时会输出：

```
Enter an integer:
```

然后立刻退出。

---

### 第 5 步：读取用户输入（先故意写错，一起看编译错误）

接下来读取用户输入。我们会用 `std::cin` 和 `operator>>` 来读，还需要一个变量来保存这个输入：

```cpp
#include <iostream>

int main() // 注意：这段代码有个错误
{
    std::cout << "Enter an integer: ";

    int num{ };        // 定义整型变量 num
    std::cin << num;   // 从键盘读入一个整数（这里写错了！）

    return 0;
}
```

编译一下……然后……

啊哦！作者在 Visual Studio 2017 上得到了下面这样的错误信息：

```
1>------ Build started: Project: Double, Configuration: Release Win32 ------
1>Double.cpp
1>c:\vcprojects\double\double.cpp(8): error C2678: binary '<<': no operator found which takes a left-hand operand of type 'std::istream' (or there is no acceptable conversion)
1>c:\vcprojects\double\double.cpp: note: could be 'built-in C++ operator<<(bool, int)'
1>c:\vcprojects\double\double.cpp: note: while trying to match the argument list '(std::istream, int)'
1>Done building project "Double.vcxproj" -- FAILED.
========== Build: 0 succeeded, 1 failed, 0 up-to-date, 0 skipped ==========
```

我们遇到了一个**编译错误**！怎么排查？

- **首先**，程序在上一步还能编译，这次改完就不行了，说明错误几乎一定在**刚加的那两行**（第 7、8 行）。这大大缩小了排查范围。第 7 行只是一个变量定义，看起来没问题，所以错误多半在**第 8 行**。
- **其次**，虽然这段错误信息不好读，但我们抓关键词：编译器说错误在**第 8 行**，说找不到一个 `<<` 运算符可以接受**左操作数是 `std::istream`**（`std::cin` 就是这个类型）。换句话说，`operator<<` 不知道该拿 `std::cin` 怎么办——所以问题要么出在 `std::cin` 用法上，要么出在 `<<` 用法上。

现在看出来了吗？如果还没看出来，先自己找一找。

修正后的代码是这样的：

```cpp
#include <iostream>

int main()
{
    std::cout << "Enter an integer: ";

    int num{ };
    std::cin >> num;   // std::cin 用的是 operator>>，不是 operator<<！

    return 0;
}
```

这下程序能编译了。运行一下，程序会等你输入一个数字，输入 `4` 后：

```
Enter an integer: 4
```

差不多快好了！最后一步：把这个数字乘以 2 输出。

---

### 第 6 步：把数字乘以 2 并输出——三种写法

至少有 **3 种**写法可以完成最后一步。我们**从最差到最好**依次看一遍。

---

#### 写法一：不推荐

```cpp
#include <iostream>

// 最差的写法
int main()
{
    std::cout << "Enter an integer: ";

    int num{ };
    std::cin >> num;

    num = num * 2; // 把 num 乘以 2，再赋回给 num 自己

    std::cout << "Double that number is: " << num << '\n';

    return 0;
}
```

这种写法用一个表达式把 `num` 乘以 2，然后把结果**赋回 `num` 自己**。之后 `num` 里存的就是“翻倍后的数”了。

为什么这种写法**不好**：

- 在赋值语句**之前**，`num` 保存的是**用户输入的原值**；赋值**之后**它保存的是一个**完全不同**的值。这**很容易让人误解**。
- 我们**用新值覆盖了用户输入**。所以如果以后想扩展程序（比如再输出“输入值的 3 倍”），**原始输入已经丢失了**。

---

#### 写法二：还不错

```cpp
#include <iostream>

// 稍微好一些的写法
int main()
{
    std::cout << "Enter an integer: ";

    int num{ };
    std::cin >> num;

    int doublenum{ num * 2 }; // 定义一个新变量，用 num * 2 来初始化
    std::cout << "Double that number is: " << doublenum << '\n';

    return 0;
}
```

这种写法**读起来清晰易懂**，也解决了上一种写法的两个问题。

主要缺点是：**为了一个只用一次的值**，我们**多定义了一个变量**（也就多了一点复杂度）。我们还能更进一步。

---

#### 写法三：推荐

```cpp
#include <iostream>

// 首选写法
int main()
{
    std::cout << "Enter an integer: ";

    int num{ };
    std::cin >> num;

    std::cout << "Double that number is: " << num * 2 << '\n'; // 在要打印的地方直接用表达式 num * 2

    return 0;
}
```

这就是这几种写法里**最优**的一版。执行 `std::cout` 时，表达式 `num * 2` 会被求值，结果就是 `num` 的两倍。这个值会被直接打印出来，而 `num` **本身的值并没有被改变**，如果之后还要用它也没问题。

这版就是我们的**参考解**。

### 作者笔记（关于“编程”这件事）

> 编程的**第一要务、也是首要目标**：**让程序跑起来**。一个**跑不起来**的程序，写得再漂亮也没用。

不过我很喜欢一句话：**“你必须先把一个程序写一遍，才知道自己一开始应该怎么写。”** 意思是：**最好的解法往往一开始并不明显，我们对问题的第一版解决方案，通常都远达不到我们能做到的最好。**

当我们的注意力全放在“怎么让程序先跑起来”时，把**大量时间**花在**还不知道是否会保留的代码**上，其实并不划算。所以我们会走捷径：**跳过错误处理、跳过注释、到处撒调试代码**，帮自己定位问题。我们会**在试错中学习**——原以为可行的方案其实不行，只好回退再试新的路子。

结果就是：**我们最初的解决方案通常结构不好、不健壮、不易读、也不够简洁**。因此，**你的程序跑起来的那一刻，工作其实还没结束**（除非它是一次性用完就扔的脚本）。**下一步是清理你的代码：**

- 删除或注释掉临时 / 调试用的代码；
- 加上注释；
- 处理错误情况；
- 调整格式；
- 遵循最佳实践。

即便做完了这些，你的程序可能仍然**不是最简洁**的写法——也许有重复的逻辑可以合并、有几条语句可以合成一条、有变量根本不需要，或者还有一千个小地方可以简化。**新手常常一味优化性能，其实更该优化的是可维护性。**

本教程里的绝大多数解法都不是“第一次写出来就完美”的，而是**反复打磨**到再也想不到能改进的地方为止。而且很多情况下，**读者还能给出更多改进建议**！

所以简单说一句：**如果你的第一版解法没有从脑子里“出场即完美”，别难过。这非常正常。编程里的“完美”，是一个不断迭代的过程（需要一遍又一遍地打磨）。**

### 作者再补一句：关于“记不住怎么办”

你可能会想：**“C++ 规则和概念这么多，我怎么可能全记住？”**

**简短答案是：你不需要全记住。** 学 C++ 的组成，大致是**“一分靠记住的东西 + 两分靠去查怎么做”**。

**第一次浏览本站教程时，不要执着于记住细节，而要重点理解“C++ 能做到什么”**。等到你要在自己写的程序里实现某个功能时，你可以随时回到这里（或其它参考网站）复习一下**具体怎么做**。
