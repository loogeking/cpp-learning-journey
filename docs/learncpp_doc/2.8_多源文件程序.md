# 2.8 — 多源文件程序

> 作者：Alex　发布时间：2025 年 2 月 11 日

## 向项目中添加文件

随着程序变大，为了**便于组织和复用**，我们通常会把代码拆分到**多个源文件**中。使用 IDE 的一个好处，就是它让**多文件开发变得轻松**。你已经知道怎么创建和编译单文件项目，接下来看看怎么把新文件加进项目里。

> **最佳实践**
>
> 向项目里新增源文件时，**用 `.cpp` 作为扩展名**。

### Visual Studio 用户

在 **Solution Explorer** 里右键 “Source Files” 文件夹（或项目名），选择 **Add → New Item…**，选中 **C++ File (.cpp)**，起个名字即可。

- 有的 VS 版本会显示简化视图，此时可以点 **“Show all Templates”** 切到完整视图。
- 如果你从 **File 菜单** 新建文件，那这个文件**不会自动加入到项目里**，需要在 Solution Explorer 中右键 “Source Files” → **Add → Existing Item**，把文件手动加进来。

编译时，你应能在编译输出中看到新文件的名字。

### Code::Blocks 用户

菜单 **File → New → File…**，在弹出的对话框里选 **C/C++ source**，一路 Next。语言选 **C++**，起个文件名（**别忘了 `.cpp` 后缀**），然后点 **All** 让所有构建目标都包含它，最后 Finish。

### gcc（命令行）用户

用你喜欢的编辑器自己创建源文件即可。编译时，把**所有相关的源文件**都加到编译命令里。例如：

```bash
g++ main.cpp add.cpp -o main
```

其中 `main.cpp` 和 `add.cpp` 是你的源文件，`main` 是输出的可执行文件名。

### VS Code 用户

- 顶部菜单 **View → Explorer** 打开资源管理器，点项目名右侧的 **New File** 图标；也可从 **File → New File** 新建，然后起一个带 `.cpp` 后缀的名字。
- 如果新文件出现在 `.vscode` 文件夹里，请把它拖回上一级的项目目录。

接下来打开 `tasks.json`，找到 `"${file}",` 这一行，有两种做法：

1. **明确列出要编译的文件**，把 `"${file}",` 换成：

    ```json
    "main.cpp",
    "add.cpp",
    ```

2. **自动编译目录下所有 `.cpp`**：
   - Windows：把 `"${file}",` 换成 `"${fileDirname}\\**.cpp"`
   - Unix / macOS：换成 `"${fileDirname}/**.cpp"`

## 一个多文件示例

在上一节 [2.7 — 前向声明与定义](#) 中我们见过这段编译不过的单文件程序：

```cpp
#include <iostream>

int main()
{
    std::cout << "The sum of 3 and 4 is: " << add(3, 4) << '\n';
    return 0;
}

int add(int x, int y)
{
    return x + y;
}
```

编译器在 `main` 的第 5 行遇到 `add` 时**根本不知道它是啥**（第 9 行才定义）！解决方案要么**调整定义顺序**，要么**加前向声明**。

现在看一个**类似但多文件**的版本：

**`add.cpp`：**

```cpp
int add(int x, int y)
{
    return x + y;
}
```

**`main.cpp`：**

```cpp
#include <iostream>

int main()
{
    std::cout << "The sum of 3 and 4 is: " << add(3, 4) << '\n'; // 编译错误
    return 0;
}
```

编译器可能先编译 `add.cpp`，也可能先编译 `main.cpp`。无论先编哪个，`main.cpp` 都会报和之前**完全相同**的错误：

```
main.cpp(5) : error C3861: 'add': identifier not found
```

**原因一模一样**：编译 `main.cpp` 到第 5 行时，编译器不知道 `add` 是什么。

**要记住**：**编译器是逐文件编译的**。它**看不到别的源文件里的内容**，也**不会记住之前编译过的文件里都有什么**。所以即使编译器之前编译 `add.cpp` 时“见过” `add` 的定义，等到编译 `main.cpp` 时它也**早就忘光了**。

这种“**视野受限、记忆力短**”是**有意为之**，好处是：

- 项目里的源文件可以**任意顺序编译**；
- 修改某个源文件时，**只需要重新编译那个文件**；
- 减少不同文件里**同名标识符冲突**的可能。

关于名字冲突，会在下一节 [2.9 — 命名冲突与命名空间简介](#) 里讨论。

**解决办法和上一节一样**：把 `add` 的定义放到 `main` 前面，或者**给编译器一个前向声明**。这里 `add` 在另一个文件里，**没法通过调整顺序解决**——所以要**用前向声明**：

**`main.cpp`（加了前向声明）：**

```cpp
#include <iostream>

int add(int x, int y); // 让 main.cpp 知道 add() 是定义在别处的函数

int main()
{
    std::cout << "The sum of 3 and 4 is: " << add(3, 4) << '\n';
    return 0;
}
```

**`add.cpp`（保持不变）：**

```cpp
int add(int x, int y)
{
    return x + y;
}
```

现在编译 `main.cpp` 时，编译器已经知道 `add` 长什么样了；**链接器**会负责把 `main.cpp` 中对 `add` 的调用，和 `add.cpp` 中 `add` 的定义**“接”上**。

用这种方式，我们就能**让一个文件访问到定义在另一个文件里的函数**。

自己动手把 `add.cpp` 和加了前向声明的 `main.cpp` 一起编译试试。如果报了链接错误，检查一下你有没有把 `add.cpp` **正确加入项目**（或加到命令行里）。

> **小贴士**
>
> 因为**编译器逐个文件编译，且不会“记住”见过什么**，所以**每个用到 `std::cout` 或 `std::cin` 的源文件都要 `#include <iostream>`**。
>
> 上面的例子里，如果 `add.cpp` 也用了 `std::cout` 或 `std::cin`，那它就也得 `#include <iostream>`。

> **关键洞察**
>
> **当某个标识符在表达式中被使用时，它必须能被“连接”到它的定义。**
>
> - 如果编译器在当前文件里**既没有见过它的前向声明、也没有见过它的定义**，就会**在使用处报编译错误**；
> - 否则，如果**当前文件里有定义**，编译器就把使用点连到那个定义；
> - 否则，如果**别的文件里有定义（且对链接器可见）**，链接器会把使用点连到那个定义；
> - 否则，链接器会**报错**，提示找不到定义。

## 出问题了怎么办？

第一次做多文件项目时容易踩坑。上面例子如果出错，检查这些：

1. **编译时报 `add` 未定义** → 多半是 `main.cpp` 里**忘了写前向声明**。

2. **链接时报 `add` 未定义**，比如：
   ```
   unresolved external symbol "int __cdecl add(int,int)" (?add@@YAHHH@Z) referenced in function _main
   ```

## 出问题了怎么办？

第一次做多文件项目时容易踩坑。上面例子如果出错，检查这些：

1. **编译时报 `add` 未定义** → 多半是 `main.cpp` 里**忘了写前向声明**。

2. **链接时报 `add` 未定义**，比如：

   ```
   unresolved external symbol "int __cdecl add(int,int)" (?add@@YAHHH@Z) referenced in function _main
   ```

   常见原因有：

   - **2a.** `add.cpp` **没被正确加入项目**。编译时你应该能在输出里同时看到 `main.cpp` 和 `add.cpp`；如果只看到 `main.cpp`，说明 `add.cpp` 根本没参与编译。用 VS 或 Code::Blocks 的话，在左/右侧的项目面板里应该能看到 `add.cpp`；看不到就右键项目把它加进来。用命令行的话，别忘了在 `g++` 命令里同时写上 `main.cpp` 和 `add.cpp`。
   - **2b.** 你可能把 `add.cpp` 加到了**错误的项目**里。
   - **2c.** 文件可能被设置成**不参与编译/链接**。查看文件属性，确认它被配置为“参与编译/链接”。在 Code::Blocks 中要检查 compile 和 link 两个复选框；在 Visual Studio 中要确认 “Exclude from build” 为 “No” 或留空，并且要**逐个构建配置**（如 Debug 和 Release）分别检查。

3. **不要在 `main.cpp` 里 `#include "add.cpp"`**。虽然在这个例子里能编译通过，但 `#include` 一个 `.cpp` 文件会**增加命名冲突和其他意外后果的风险**（程序越大越明显）。关于 `#include`，我们会在 [2.10 — 预处理器简介](#) 里进一步讨论。

   - **2b.** 你可能把 `add.cpp` 加到了**错误的项目**里。
   - **2c.** 文件可能被设置成**不参与编译/链接**。查看文件属性，确认它被配置为“参与编译/链接”。在 Code::Blocks 中要检查 compile 和 link 两个复选框；在 Visual Studio 中要确认 “Exclude from build” 为 “No” 或留空，并且要**逐个构建配置**（如 Debug 和 Release）分别检查。
