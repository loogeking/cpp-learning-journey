# 2.13 — 如何设计你的第一个程序

---

现在你已经掌握了一些程序的基础知识，让我们把镜头再拉近一点，看看**怎么去设计一个程序**。

当你坐下来准备写程序时，你脑子里通常已经有了某个**想法**，你想把它做出来。**新手常常卡在“怎么把这个想法变成实际代码”这一步**。但其实，你早已从日常生活中获得了很多解决问题所需的能力。

**最重要（也是最难做到的）的一件事是：在动手写代码之前先做设计。**

在很多方面，编程和建筑很像。你想没想过——**不看图纸就直接盖房子**会怎么样？除非你是个超级天才，否则墙不会直、屋顶会漏水，各种问题一大堆。写程序也是一样：**如果你没有明确的思路就急着敲代码，代码里往往会有一堆本可以提前避免的问题**，你只能事后花大量时间去修。

**先花一点时间做计划，长远来看能省下大量时间和挫败感。**

本节我们给你一个**把想法变成简单可用程序**的通用方法。

---

## 设计第 1 步：明确你的目标

要写出一个成功的程序，第一步就是**明确你要达成的目标**。理想情况下，能用一两句话概括。**用“用户视角的结果”来表述**通常最有用。例如：

- 让用户能组织一个包含姓名和电话号码的名单。
- 生成随机的地下城，形成有趣的洞穴地图。
- 根据高分红特点，生成一份股票推荐清单。
- 建模一个从高塔掉落的球到达地面需要的时间。

这一步看起来很显然，但**极其重要**。**最糟的事情就是**：你写了一个程序，**它做的事根本不是你（或你老板）想要的**。

---

## 设计第 2 步：明确“需求”

“明确问题”帮你确定**想要什么结果**，但还是很模糊。下一步是**思考需求（requirements）**。

**“需求”**这个词，包含两方面内容：

1. 解决方案必须遵守的**约束**（比如预算、时间、空间、内存等）；
2. 程序为满足用户需要而必须具备的**能力**。

同样地，**需求要聚焦“做什么”，而不是“怎么做”**。例如：

- 电话号码要能被保存，以便日后调用。
- 随机地下城必须**保证存在从入口到出口的路径**。
- 股票推荐应利用**历史价格数据**。
- 用户应能**输入塔的高度**。
- **7 天内**要有一个可测试版本。
- 程序应在**用户提交请求后 10 秒内**给出结果。
- 程序在用户会话中崩溃的比例应**小于 0.1%**。

**一个问题往往会衍生出很多需求，只有全部满足，方案才算“完成”。**

---

## 设计第 3 步：明确你的工具、目标平台与备份方案

如果你是有经验的程序员，这一步通常还会包含：

- 明确程序运行的**目标架构 / 操作系统**；
- 决定要使用的**工具链**；
- 决定是**单打独斗**还是**团队协作**；
- 明确**测试 / 反馈 / 发布**策略；
- 决定**代码如何备份**。

但对新手来说，答案通常都很简单：**为自己写、一个人做、在自己的电脑上，用下载来的 IDE，代码基本只有自己用**。这就轻松多了。

尽管如此，**只要你在做任何“不是玩具级”的项目，都应该有备份方案**：

- **仅仅**把源码目录压缩/复制到**同一台设备的另一个目录**是不够的——**硬盘坏了、损坏了，一切都会没**；
- 拷贝或压缩到**移动存储**（比如 U 盘）会好一些，但**丢失、被盗、火灾或重大自然灾害**仍会让你血本无归。

**最佳的备份策略是——让代码在“物理位置不同”的机器上也存在一份**。做法很多且简单：

- 压缩后邮件发给自己；
- 上传到云盘（比如 Dropbox）；
- 用 SFTP 之类的协议上传到你控制的服务器；
- 使用位于**另一台机器或云端**的**版本控制系统**（如 GitHub）。

**版本控制系统还有额外好处**：不仅能恢复文件，还能**回滚到之前的版本**。

---

## 设计第 4 步：把难问题拆成简单问题

生活中我们经常面对非常复杂的任务，直接想“怎么做”会很头大。这时通常用 **“自顶向下（top-down）”** 的思路：**先把大任务拆成若干小任务，再把每个小任务拆得更细**，直到每一步都简单到可以直接完成。

来个例子。假设我们要“**打扫房子**”：

- 打扫房子

一次性搞定太大，拆一下：

- 打扫房子
  - 吸尘（清理地毯）
  - 打扫卫生间
  - 打扫厨房

再进一步细化：

- 打扫房子
  - 吸尘
  - 打扫卫生间
    - 刷马桶（呃……）
    - 洗洗手台
  - 打扫厨房
    - 清空台面
    - 擦干净台面
    - 洗水槽
    - 倒垃圾

这样每个子任务都不太难。**逐个完成这些小任务，就完成了“打扫房子”这个大任务。**

**另一种做法是自底向上（bottom-up）**：**先列出一堆简单任务，再把相似的归类，形成层级**。

比如“**去上班**”。你可能列出：

- 挑衣服
- 穿衣服
- 吃早餐
- 上路去公司
- 刷牙
- 起床
- 准备早餐
- 骑上自行车
- 洗澡

自底向上组织成：

- 从床到公司
  - 卧室相关
    - 关闹钟
    - 起床
    - 挑衣服
  - 卫生间相关
    - 洗澡
    - 穿衣服
    - 刷牙
  - 早餐相关
    - 泡咖啡/茶
    - 吃麦片
  - 交通相关
    - 骑上自行车
    - 去公司

事实证明，**这种“任务树”对编程非常有用**。因为一旦你有了这样的层级：

- **最顶层的大任务**就是 `main()`（这是你要解决的核心问题）；
- **子任务**就是程序里的**函数**。

如果某个函数**太难**实现，就把它继续拆成更小的子函数。**拆到每个函数都简单为止。**

---

## 设计第 5 步：确定“事件顺序”

程序结构有了之后，就要**决定这些任务的先后顺序**。比如“早上起床后你做事情的顺序”是这样的：

1. 卧室相关
2. 卫生间相关
3. 早餐相关
4. 交通相关

如果我们要写一个**计算器**，顺序可能是：

1. 从用户获取第一个数
2. 从用户获取运算符
3. 从用户获取第二个数
4. 计算结果
5. 打印结果

到这里，就可以开始**实现**了。

---

## 实现第 1 步：先把 `main` 的“骨架”搭出来

有了上面的事件顺序，就可以用它来搭 `main` 的大纲。**暂时不要纠结输入输出**。

```cpp
int main()
{
//    doBedroomThings();
//    doBathroomThings();
//    doBreakfastThings();
//    doTransportationThings();

    return 0;
}
```

计算器：

```cpp
int main()
{
    // 获取第一个数
//    getUserInput();

    // 获取运算符
//    getMathematicalOperation();

    // 获取第二个数
//    getUserInput();

    // 计算结果
//    calculateResult();

    // 输出结果
//    printResult();

    return 0;
}
```

**注意**：用这种“先写大纲”的方式，如果你把函数调用**没注释掉**，程序就会因为“函数还没定义”而编不过。**两种解决办法**：

- **像上面这样先把调用注释掉**，等实现了再打开（本教程就这么做）；
- 或者**先写函数“桩（stub）”**：给出函数原型并写一个空的函数体，让程序先能编译过。

---

## 实现第 2 步：逐个实现每个函数

对每个函数，都做三件事：

1. **定义函数原型**（输入是什么、输出是什么）；
2. **写出函数体**；
3. **测试这个函数**。

**如果函数拆得足够细**，每一个都应该相对简单直白。如果实现某个函数时**依然感觉很复杂**，可能需要**再拆一层**；也可能是你**顺序设计得不合适**，需要回头调整。

来实现计算器的第一个函数：

```cpp
#include <iostream>

// getUserInput 的完整实现
int getUserInput()
{
    std::cout << "Enter an integer: ";
    int input{};
    std::cin >> input;

    return input;
}

int main()
{
    // 获取第一个数
    int value{ getUserInput() }; // 注意这里我们写了代码来“测试返回值”！
    std::cout << value << '\n';  // 调试用，稍后会删掉

    // 获取运算符
//    getMathematicalOperation();

    // 获取第二个数
//    getUserInput();

    // 计算结果
//    calculateResult();

    // 输出结果
//    printResult();

    return 0;
}
```

我们先确定 `getUserInput` **不需要参数**、返回 `int`，这在函数原型里就体现出来了。然后写了简单的 4 条语句作为函数体。最后在 `main` 里加了**临时测试代码**来验证 `getUserInput`（以及它的返回值）能否正常工作。

我们可以**用不同的输入多次运行**这个程序，确保当前行为符合预期。**如果出问题了，我们知道 bug 一定在我们刚写的这段代码里。**

一旦确认当前部分工作正常，就可以**删掉临时测试代码**，去实现下一个函数（比如 `getMathematicalOperation`）。本节我们不会把整个程序写完，因为还需要先学一些别的内容。

**记住：不要一口气把整个程序写完。**
**分步实现，每一步都测试，通过后再往下走。**

## 实现第 3 步：最终整体测试

程序“看起来写完”后，**最后一步就是整体测试**，确保它按预期工作。**如果不对，就修。**

## 写程序时的一些建议

- **一开始把程序做简单**。新手常常一上来就想做超大的东西：“我要写一个带图形和音效、随机怪物、随机地下城、还有城镇卖装备的角色扮演游戏。” 这么大的目标会让你**很快就被劝退**。**先把目标定得非常小、绝对做得到**，比如“**能在屏幕上显示一个二维地图**”就够了。

- **逐步加功能**。等最基础的程序能跑得很稳时，再一点点添功能：能显示地图之后，加一个可以走动的角色；能走动之后，加一些墙；有墙之后，用它们搭个小镇；有小镇之后，再加商人……**每次只加一小步**，程序会稳步变复杂，而不会把你压垮。

- **一次只专注一件事**。**别想把所有事情同时写完**，也别把注意力分散到多个任务上。**5 个任务没开始、1 个任务已经跑通** 远远好过 **6 个任务都做了一半**。分心会让你更容易犯错、漏掉细节。

- **写一段测一段**。新手很爱一口气把整个程序敲完，然后编译一下——蹦出来几百个错误。这**又打击人**、又难定位。**正确姿势**是：**写一小段 → 立刻编译 → 立刻测试**。有问题的话，你**清清楚楚**知道问题就在刚写的这段。全部通过后再写下一段。**这样看似写得慢，其实整体比“最后调 bug 调到吐”要快得多。**

- **别在早期代码上追求完美**。功能（或程序）的**第一稿几乎不会好**。而且程序会随着能力增加**不断演化**。**如果一开始就投入太多精力去打磨（加大量文档、严格套所有最佳实践、微观优化）**，一旦需要改动，之前的投入就白费了。**先做到功能能跑，再往前走**；等对方案越来越有信心后，**再一层一层地打磨**。**别追求完美**——**不平凡的程序永远都不完美**，总能再改得更好。**做到“足够好”，然后前进。**

- **优先优化“可维护性”，而不是“性能”**。有一句名言（Donald Knuth）：**“过早优化是万恶之源”**。新手常在“哪两条语句更快”这种问题上耗掉大量时间——**这些几乎都不重要**。**真正的性能收益**大多来自：**良好的结构、恰当的工具与能力、遵循最佳实践**。多余的时间应该拿去**提升可维护性**：**消除重复**、**把长函数拆短**、**用更清晰的代码替换难懂的代码**。这样以后**优化更容易**（当你真的定位到瓶颈时），**bug 也更少**。更多建议见 [3.10 — 在问题变成 bug 之前发现它们](#)。

> **一个能工作的复杂系统，无一例外都是从一个能工作的简单系统演化而来的。**
> —— John Gall，《Systemantics: How Systems Really Work and How They Fail》第 71 页

---

## 小结

许多新手会**跳过设计阶段**（觉得麻烦，也没写代码好玩）。但对于任何**不是玩具级**的项目，**遵循上面这些步骤会为你省下大量时间**。**前期一点点计划，胜过后期一大堆调 bug。**

> **关键洞察**
>
> **在结构上多花一点点时间，会让你的代码更好，也会让你在“找错、修错”上花更少的时间。**

> **我觉得这可能是编程中最重要的一件事，而我们中的一些人（包括我自己）一开始都低估了它。**
> —— 读者 Emeka Daniel 在 learncpp.com 上的评论

当你越来越熟练地掌握这些概念和技巧后，它们会**变成本能**。最终你会达到这样的水平：**只需极少的事先设计，就能写出完整的函数（甚至简短的程序）**。
