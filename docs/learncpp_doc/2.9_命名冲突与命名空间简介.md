# 2.9 — 命名冲突与命名空间简介

---

设想一下：你第一次开车去朋友家，地址是 **Mill City, Front Street 245 号**。到了 Mill City 你掏出地图，结果发现——**这个城市里居然有两条 Front Street，还在城市两头**！你该去哪一条？除非有别的线索（比如你记得朋友家靠河），否则你只能打电话问朋友。为了避免这种混乱和低效（尤其对送信的邮差来说是灾难），**大多数国家都会规定：一个城市内的所有街道名和门牌号必须唯一。**

C++ 的要求也类似：**所有标识符必须无歧义**。如果两个**完全相同的标识符**以一种编译器或链接器**分不清楚**的方式出现在同一个程序里，就会产生错误。这种错误一般叫做 **命名冲突（naming collision / naming conflict）**。

- **冲突发生在同一个文件里** → **编译错误**；
- **冲突发生在同一个程序的不同文件里** → **链接错误**。

---

## 一个命名冲突的例子

**`a.cpp`**：

```cpp
#include <iostream>

void myFcn(int x)
{
    std::cout << x;
}
```

**`main.cpp`**：

```cpp
#include <iostream>

void myFcn(int x)
{
    std::cout << 2 * x;
}

int main()
{
    return 0;
}
```

编译器会**独立地**编译 `a.cpp` 和 `main.cpp`，**这一步两个文件都不会报错**。

但轮到**链接器**工作时，它会把 `a.cpp` 和 `main.cpp` 中的所有定义放到一起，发现 `myFcn()` 有**两个冲突的定义**，于是**报错并终止链接**。**注意这个错误即便 `myFcn()` 从未被调用也照样会发生！**

**最常见的命名冲突有两种：**

1. **两个（或更多）同名函数或全局变量** 被引入到**同一程序的不同文件**中 → **链接错误**（就是上面这个例子）。
2. **两个（或更多）同名函数或全局变量** 被引入到**同一个文件**中 → **编译错误**。

程序越大、用到的标识符越多，出现命名冲突的可能性就会显著增加。**好消息是**：C++ 提供了很多避免命名冲突的机制。**局部作用域（local scope）** 就是其中之一——它让不同函数里的局部变量互不干扰。但局部作用域**帮不了函数名**：那**函数名之间的冲突要靠什么解决呢**？

---

## 作用域区域（Scope regions）

回到刚才的地址类比：**两个 Front Street 会打架，是因为它们都在同一座城市里**。如果一个 Front Street 245 号在 Mill City、另一个在 Jonesville，那就完全不会混淆了。换句话说，**“城市”提供了一种“分组”，帮我们区分开原本会冲突的地址**。

**作用域区域（scope region）** 就是源码中的一片区域：**在其内声明的所有标识符，与其它作用域区域中声明的同名标识符是相互独立的**（就像上面例子里的“城市”）。因此，**同名的标识符如果分别声明在不同的作用域区域中，是不会冲突的**；但在**同一个作用域区域内**，所有标识符**必须唯一**，否则就会发生命名冲突。

**函数体就是一种作用域区域**。所以在两个不同函数里定义同名的局部标识符，是完全没问题的——因为它们分别属于两个不同的作用域区域。但如果你在同一个函数里定义两个同名标识符，就会产生命名冲突，编译器会报错。

---

## 命名空间（Namespaces）

**命名空间（namespace）** 提供了**另一种作用域区域**（叫做**命名空间作用域 namespace scope**），让你在其中声明或定义名字，从而实现**名称的“消歧义”**。**命名空间中声明的名字，与其他作用域中声明的名字彼此隔离**，不会冲突。

> **关键洞察**
>
> 在一个作用域区域（例如命名空间）中声明的名字，与其它作用域中声明的同名标识符是**互不相同**的。

例如，两个**声明完全一样**的函数分别定义在**不同的命名空间**中，就**不会**发生命名冲突或歧义。

**命名空间只能包含声明和定义**（例如变量和函数）。**可执行语句是不允许直接放在命名空间里的**，除非它们是某个定义的一部分（比如函数体内）。

> **关键洞察**
>
> **命名空间只能包含声明和定义。** 可执行语句只允许出现在某个定义（例如某个函数）的内部。

在大型项目中，我们经常用命名空间来**把相关的标识符组织在一起**，防止它们不小心和别的标识符冲突。例如，如果你把所有数学函数都放到叫 `math` 的命名空间中，那你 `math` 里的函数就不会和 `math` 之外同名的函数打架。

至于**如何创建自己的命名空间**，会在后续课程中讲。

---

## 全局命名空间（The global namespace）

在 C++ 中，**任何没有定义在类、函数或某个命名空间中的名字**，都被视为属于一个**隐式定义的命名空间——全局命名空间（global namespace）**（有时也叫**全局作用域**）。

在本节开头的例子里，`main()` 和两个版本的 `myFcn()` 都是定义在**全局命名空间**里的。命名冲突之所以发生，就是因为两个 `myFcn()` **都落在了同一个全局命名空间中**，违反了“同一作用域中所有名字必须唯一”的规则。

关于全局命名空间的更多内容，会在 [7.4 — 全局变量简介](#) 中讨论。目前，你只需要知道两点：

- **在全局作用域中声明的标识符**，其作用域从**声明处**一直延伸到**文件末尾**；
- 变量**可以**定义在全局命名空间中，但**通常应避免这么做**（原因见 [7.8 — 为什么（非常量）全局变量是万恶之源](#)）。

例如：

```cpp
#include <iostream> // 把 std::cout 的声明引入全局作用域

// 下面所有语句都属于全局命名空间

void foo();    // OK：函数前向声明
int x;         // 能编译，但强烈不推荐：无初始化器的非常量全局变量
int y { 5 };   // 能编译，但强烈不推荐：带初始化器的非常量全局变量
x = 5;         // 编译错误：命名空间中不允许出现可执行语句

int main()     // OK：函数定义
{
    return 0;
}

void goo();    // OK：函数前向声明
```

## `std` 命名空间

C++ 最初被设计出来时，标准库中的所有标识符（包括 `std::cin`、`std::cout`）都**可以直接使用，不需要 `std::` 前缀**（它们都属于全局命名空间）。但这样一来，标准库里的任何一个名字都**可能与你自己起的名字冲突**（也定义在全局命名空间）。原本能正常工作的代码，**只要引入了标准库的某个部分，就可能突然冲突**；更糟的是，你的代码可能在旧版 C++ 上编得过，**在新版 C++ 里就编不过**——因为新版标准库可能新增了一些名字，恰好和你原有代码里的名字冲突。

于是 C++ 把标准库里的所有内容都**移到了一个叫 `std`（standard 的缩写）的命名空间里**。

事实上，`std::cout` 的**真正名字**其实**就是 `cout`**，而 `std` 是**这个 `cout` 所在的命名空间的名字**。因为 `cout` 定义在 `std` 命名空间中，所以它**不会**和你在 `std` 之外（例如全局命名空间中）定义的、名字叫 `cout` 的对象或函数冲突。

> **关键洞察**
>
> **当你使用的标识符定义在一个非全局的命名空间（例如 `std`）中时，你需要告诉编译器：这个标识符位于那个命名空间里。**

有几种做法。

---

## 显式命名空间限定：`std::`

最直接的方式就是**使用 `std::` 前缀显式指定**：

```cpp
#include <iostream>

int main()
{
    std::cout << "Hello world!"; // 我们说 cout 时，指的是 std 命名空间里的那个 cout
    return 0;
}
```

`::` 是一个运算符，叫做 **作用域解析运算符（scope resolution operator）**。它**左边**的标识符表示命名空间，**右边**是这个命名空间里的名字。如果 `::` 左边什么都不写，则默认是**全局命名空间**。

因此，`std::cout` 的意思就是 **“`std` 命名空间中声明的那个 `cout`”**。

这是最安全的用法，因为**没有任何歧义**：我们明确指的就是 `std` 命名空间里那个 `cout`。

> **最佳实践**
>
> **使用显式的命名空间前缀去访问命名空间中的标识符。**

带命名空间前缀的标识符也被称为 **限定名（qualified name）**。

---

## `using namespace std;`（以及为什么要避免它）

访问命名空间中标识符的另一种方式，是使用 **using-directive（using 指令）**。下面是加了 using 指令的“Hello world”程序：

```cpp
#include <iostream>

using namespace std; // 这是一个 using 指令：允许我们无前缀访问 std 里的名字

int main()
{
    cout << "Hello world!";
    return 0;
}
```

**using 指令**允许我们**不加前缀地**访问某个命名空间中的名字。所以上例中，编译器碰到 `cout` 时，会把它匹配到 `std::cout`——因为 `using namespace std;` 的存在，它现在可以直接写作 `cout`。

**很多教材、教程、甚至一些 IDE 都会在程序开头加一句 `using namespace std;`**。但**这样用其实是很糟糕的做法**，**非常不推荐**。

看下面这段程序：

```cpp
#include <iostream> // 把 std::cout 的声明引入全局作用域

using namespace std; // 让 std::cout 可以直接写作 cout

int cout() // 我们又在全局命名空间里定义了一个自己的 cout 函数
{
    return 5;
}

int main()
{
    cout << "Hello, world!"; // 编译错误！这里的 cout 到底是哪一个？
                              // 是 std 里的那个，还是我们上面定义的这个？
    return 0;
}
```

这段程序编译不过，因为编译器**无法判断你要的是 `std::cout` 还是我们自己定义的那个 `cout` 函数**。

**只要你这样使用 using 指令，你定义的任何名字都可能和 `std` 中的同名标识符冲突。** 更糟的是：即便**今天不冲突**，将来 C++ 标准往 `std` 里**新增**了同名标识符，你的代码可能**未来就编不过**了——而这**恰恰**是当初把整个标准库塞进 `std` 命名空间要避免的事情！

> **警告**
>
> **不要在程序顶部或头文件里使用 using 指令**（比如 `using namespace std;`）。**它违背了命名空间被引入语言的初衷。**

> **相关内容**
>
> 关于 **using 声明** 与 **using 指令**（以及如何“负责任地”使用它们），我们会在 [7.13 — using 声明与 using 指令](#) 中进一步讨论。

## 大括号与代码缩进

在 C++ 中，**大括号常用来划分一个嵌套在另一个作用域区域内部的作用域区域**（大括号也用于一些**与作用域无关**的用途，比如**列表初始化**）。例如，一个定义在全局作用域里的函数，就用大括号把**函数体的作用域**从**外部的全局作用域**中划分出来。

在某些情况下，**定义在大括号之外的标识符**也可能属于大括号所划分出来的那个作用域，而不是外层作用域——**函数的形参**就是一个很好的例子。

例如：

```cpp
#include <iostream> // 把 std::cout 的声明引入全局作用域

void foo(int x) // foo 定义在全局作用域，x 定义在 foo() 的作用域内
{ // 用大括号划分出 foo() 的嵌套作用域
    std::cout << x << '\n';
} // x 在这里离开作用域

int main()
{ // 用大括号划分出 main() 的嵌套作用域
    foo(5);

    int x { 6 }; // x 定义在 main() 的作用域内
    std::cout << x << '\n';

    return 0;
} // x 在这里离开作用域
// foo、main（以及 std::cout）在文件末尾离开作用域
```

**嵌套作用域中的代码，按惯例会缩进一级**——既方便阅读，也直观地提示读者：**这段代码位于一个更内层的作用域中**。

`#include` 以及 `foo()`、`main()` 的函数定义都位于**全局作用域**，因此不缩进；而这两个函数**内部的语句**都位于**函数的嵌套作用域**中，所以缩进一级。
