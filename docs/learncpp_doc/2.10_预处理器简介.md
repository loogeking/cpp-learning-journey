# 2.10 — 预处理器简介

---

编译项目时，你可能以为编译器会**原封不动地**按你写的样子来编译每个源文件。**其实并不是这样。**

**在真正编译之前，每个 `.cpp` 源文件都会先经过一个“预处理阶段”。** 这个阶段里，有一个叫做**预处理器（preprocessor）** 的程序，会对源文件的**文本**做一系列改动。**注意：预处理器不会修改你原本的源文件**——它做的所有改动，要么发生在**内存中的临时副本**上，要么发生在**临时文件**里。

> **顺便一提**
>
> 历史上，预处理器是**独立于编译器的一个程序**；在现代编译器中，预处理器**往往已经内置在编译器里**了。

预处理器做的大部分事情其实**挺无聊**：比如**剥离注释**、**确保每个源文件以换行符结尾**等等。但它有一个**非常重要**的职责：**处理 `#include` 指令**（我们马上会讲到）。

预处理器处理完一个源文件后，得到的结果被称为一个 **翻译单元（translation unit）**。**编译器真正编译的，就是这个翻译单元**，而不是你原始的源文件。

> **相关内容**
>
> 从预处理、编译到链接的整个过程，统称为 **翻译（translation）**。
>
> 如果你感兴趣，可以去查一下 C++ 的“翻译阶段”列表。目前来说：**预处理对应阶段 1~4，编译对应阶段 5~7**。

---

## 预处理器指令（Preprocessor directives）

预处理器工作时，会从上到下扫描源文件，寻找**预处理器指令**。**预处理器指令**（通常简称为 **directive**）以 `#` 开头，以**换行**结尾（**不是**分号）。这些指令告诉预处理器要做哪些**文本层面的处理**。

要特别注意：**预处理器并不理解 C++ 语法**——它有自己的一套语法，有些看起来像 C++，有些则完全不像。

> **关键洞察**
>
> **预处理器最终输出的内容里不会再有任何指令**——**只有指令处理后的结果**才会被交给编译器。

> **顺便一提**
>
> 上一节讲到的 **using directive**（`using namespace ...;`）**并不是**预处理器指令，所以**不会**被预处理器处理。虽然 “directive” 一般指“预处理器指令”，但**并不总是如此**。

---

## `#include`

你已经见过 `#include` 了（一般是 `#include <iostream>`）。当你 `#include` 一个文件时，**预处理器会用被包含文件的内容替换 `#include` 那一行**。被替换进来的内容也会继续被预处理（可能会**递归**地处理其中的 `#include`），然后再继续处理剩下的部分。

例如：

```cpp
#include <iostream>

int main()
{
    std::cout << "Hello, world!\n";
    return 0;
}
```

预处理器会**用 `iostream` 文件的内容**替换掉 `#include <iostream>`，然后继续预处理这些被包含进来的内容以及剩余的代码。

因为 `#include` 几乎**只用来包含头文件**，我们会在下一节详细讨论它。

> **关键洞察**
>
> **一个翻译单元通常由一个 `.cpp` 源文件 + 它 `#include` 的所有头文件构成**（`#include` 是递归的，因为头文件里可能又 `#include` 别的头文件）。

---

## 宏定义（Macro defines）

`#define` 指令可以用来**创建一个宏（macro）**。在 C++ 中，**宏就是一条规则**，它告诉预处理器：**如何把输入文本替换成另一段文本**。

宏基本分两种：

- **对象式宏（object-like macros）**
- **函数式宏（function-like macros）**

**函数式宏用起来像函数**，作用也类似。但它们**通常被视为不安全**，几乎所有它们能做的事，用**普通函数**都能做，因此**不建议使用**。

**对象式宏**有两种定义形式：

```cpp
#define IDENTIFIER
#define IDENTIFIER substitution_text
```

上面这种**没有替换文本**，下面这种**有替换文本**。**注意：它们都是预处理器指令而不是语句，所以末尾都不加分号。**

宏名的命名规则和普通标识符相同：由字母、数字、下划线组成，不能以数字开头，也**不建议以下划线开头**。**按惯例，宏名全部大写，单词之间用下划线分隔。**

> **最佳实践**
>
> **宏名应全部大写，单词之间用下划线分隔。**

---

## 带替换文本的对象式宏

预处理器遇到这类指令时，会**把宏名和替换文本关联起来**。之后，**除了在其他预处理器指令中出现的情况外**，源码中所有出现的这个宏名都会被替换成替换文本。

例如：

```cpp
#include <iostream>

#define MY_NAME "Alex"

int main()
{
    std::cout << "My name is: " << MY_NAME << '\n';

    return 0;
}
```

预处理器会把上面这段转换成：

```cpp
// iostream 的内容会被插到这里

int main()
{
    std::cout << "My name is: " << "Alex" << '\n';

    return 0;
}
```

运行后输出：`My name is: Alex`。

**在 C 语言里**，人们经常用“带替换文本的对象式宏”来给字面量起名字。但在 **C++** 中已经**不再需要**这么做了——C++ 提供了更好的做法（见 [7.10 — 用内联变量在多文件间共享全局常量](#)）。**如今这种宏主要出现在遗留代码里，能不用就不用。**

> **最佳实践**
>
> **除非确实没有更好的替代方案，否则尽量避免使用带替换文本的宏。**

---

## 不带替换文本的对象式宏

对象式宏也可以**没有替换文本**：

```cpp
#define USE_YEN
```

这种宏正如你想的那样：**源码中大多数出现的这个标识符会被“替换成什么都没有”**！

看起来毫无用处？确实，**它对“文本替换”没什么用**。但**这类宏并不是用来做替换的**——它有**另外的用途**，我们马上就会看到。

**和带替换文本的宏不同**，这种“没有替换文本的宏”**一般被认为是可以接受使用的**。

## 条件编译（Conditional compilation）

**条件编译指令**允许你指定：**满足什么条件时**，某段代码**参与编译**，**否则不参与编译**。条件编译指令有很多，本节只讲最常用的 3 个：`#ifdef`、`#ifndef`、`#endif`。

`#ifdef` 让预处理器检查某个标识符**是否已经通过 `#define` 定义过**。**如果定义过**，`#ifdef` 和对应的 `#endif` 之间的代码会**参与编译**；**否则**，那段代码会**被忽略**。

例如：

```cpp
#include <iostream>

#define PRINT_JOE

int main()
{
#ifdef PRINT_JOE
    std::cout << "Joe\n"; // 会被编译，因为 PRINT_JOE 已定义
#endif

#ifdef PRINT_BOB
    std::cout << "Bob\n"; // 会被剔除，因为 PRINT_BOB 未定义
#endif

    return 0;
}
```

由于 `PRINT_JOE` 被 `#define` 了，`std::cout << "Joe\n";` 会被编译；`PRINT_BOB` 没被 `#define`，所以 `std::cout << "Bob\n";` 会被忽略。

`#ifndef` 是 `#ifdef` 的**反义词**——它检查某个标识符是否**尚未**被 `#define`：

```cpp
#include <iostream>

int main()
{
#ifndef PRINT_BOB
    std::cout << "Bob\n";
#endif

    return 0;
}
```

这个程序会输出 `Bob`，因为 `PRINT_BOB` 从未被 `#define`。

除了 `#ifdef PRINT_BOB` 和 `#ifndef PRINT_BOB`，你也会看到 `#if defined(PRINT_BOB)` 和 `#if !defined(PRINT_BOB)`，效果一样，只是写法更**偏 C++ 风格**。

---

## `#if 0`

条件编译还有一个很常见的用法：**用 `#if 0` 把一段代码从编译中排除**（就像被包在一个注释里一样）：

```cpp
#include <iostream>

int main()
{
    std::cout << "Joe\n";

#if 0 // 从这里开始不编译
    std::cout << "Bob\n";
    std::cout << "Steve\n";
#endif // 一直排除到这里

    return 0;
}
```

上面这段程序只会输出 `Joe`，因为 `Bob` 和 `Steve` 那部分被 `#if 0` 排除掉了。

这也**非常方便地解决了“注释掉一段本身就包含多行注释的代码”的问题**——因为多行注释**不能嵌套**，直接用 `/* ... */` 是不行的：

```cpp
#include <iostream>

int main()
{
    std::cout << "Joe\n";

#if 0 // 从这里开始不编译
    std::cout << "Bob\n";
    /* 一段
     * 多行
     * 注释
     */
    std::cout << "Steve\n";
#endif // 一直排除到这里

    return 0;
}
```

如果想**临时把被 `#if 0` 屏蔽的代码启用回来**，只需把 `#if 0` 改成 `#if 1`：

```cpp
#include <iostream>

int main()
{
    std::cout << "Joe\n";

#if 1 // 恒为真，下面这段会被编译
    std::cout << "Bob\n";
    /* 一段
     * 多行
     * 注释
     */
    std::cout << "Steve\n";
#endif

    return 0;
}
```

## 宏在其它预处理器指令中的替换问题

看了上面这段代码你可能会疑惑：

```cpp
#define PRINT_JOE

int main()
{
#ifdef PRINT_JOE
    std::cout << "Joe\n"; // 会被编译，因为 PRINT_JOE 已被定义
#endif

    return 0;
}
```

我们把 `PRINT_JOE` 定义成了“什么都没有”，那为什么预处理器**不会**把 `#ifdef PRINT_JOE` 里的 `PRINT_JOE` 也替换成“什么都没有”，从而把那段输出剔除掉？

**因为大多数情况下：当宏名出现在另一个预处理器指令里时，宏替换不会发生。**

> **顺便一提**
>
> 这条规则至少有**一个例外**：**大多数形式的 `#if` 和 `#elif` 会在预处理器指令内部进行宏替换。**

再看一个例子：

```cpp
#define FOO 9 // 这里是一个宏定义

#ifdef FOO // 这里的 FOO 不会被替换成 9，因为它属于另一个预处理器指令的一部分
    std::cout << FOO << '\n'; // 这里的 FOO 会被替换成 9，因为它属于普通代码
#endif
```

## `#define` 的作用范围（Scope）

**指令是在编译之前、按“逐文件、从上到下”的顺序**处理的。

看下面这段程序：

```cpp
#include <iostream>

void foo()
{
#define MY_NAME "Alex"
}

int main()
{
    std::cout << "My name is: " << MY_NAME << '\n';

    return 0;
}
```

尽管看起来 `#define MY_NAME "Alex"` **写在了函数 `foo` 内部**，但**预处理器根本不理解“函数”这种 C++ 概念**。因此这段程序**等价于**把 `#define MY_NAME "Alex"` 写在 `foo` **之前或紧接其后**的效果。为了避免混淆，**通常应把 `#define` 写在函数之外**。

因为 `#include` 会**把被包含文件的内容“搬运”**到当前位置，所以 `#include` 可以**把被包含文件里的指令也带到当前文件里来**，然后按顺序处理。

例如，下面这段效果和前面一样：

**`Alex.h`**：

```cpp
#define MY_NAME "Alex"
```

**`main.cpp`**：

```cpp
#include "Alex.h" // 相当于把 #define MY_NAME "Alex" 复制到这里
#include <iostream>

int main()
{
    std::cout << "My name is: " << MY_NAME << '\n'; // MY_NAME 被替换成 "Alex"

    return 0;
}
```

**预处理器一旦处理完某个文件，所有在该文件中定义的标识符就会被“丢弃”。** 也就是说，**指令的有效范围**是**从定义处一直到它所在文件的末尾**。**一个文件中的指令不会影响其它文件**（除非那个文件通过 `#include` 把它引进来）。

例如：

**`function.cpp`**：

```cpp
#include <iostream>

void doSomething()
{
#ifdef PRINT
    std::cout << "Printing!\n";
#endif
#ifndef PRINT
    std::cout << "Not printing!\n";
#endif
}
```

**`main.cpp`**：

```cpp
void doSomething(); // doSomething() 的前向声明

#define PRINT

int main()
{
    doSomething();

    return 0;
}
```

程序运行输出：

```
Not printing!
```

尽管 `PRINT` 在 `main.cpp` 中被定义了，但**它对 `function.cpp` 中的代码没有任何影响**（`PRINT` 的作用范围仅限于 `main.cpp` 中定义处到文件末尾）。等我们讲到 **头文件保护（header guards）** 时，你会看到这一点非常关键。
