# 2.11 — 头文件

---

在 [2.8 — 多源文件程序](#) 中，我们讨论了如何把程序拆到多个源文件里，以及如何用**前向声明**让一个文件里的代码访问定义在另一个文件里的东西。

程序小、文件少的时候，手动在每个文件顶部加几行前向声明还能忍。但**程序一旦变大**（文件更多、函数更多），要在**每个文件**顶部手动维护**一大堆（还可能各不相同的）前向声明**，就变得**极其枯燥**。想象一下：5 个文件、每个需要 10 个前向声明 → 你要复制/粘贴 50 条前向声明；100 个文件、各需 100 条 → 完全不可扩展！

为了解决这个问题，C++ 程序通常采用**另一种做法**。

---

## 头文件（Header files）

C++ 程序里，除了 `.cpp` 源文件，还有一种常见的文件叫做 **头文件（header file）**。头文件通常以 **`.h`** 为扩展名，偶尔也会看到 `.hpp`，或者干脆没有扩展名。

按惯例，**头文件用来把一批相关的前向声明“传播”到源文件里**。

> **关键洞察**
>
> 头文件让我们能**把声明集中放在一处，然后在需要它们的地方“引入”过来**。这在多文件项目中能节省大量重复劳动。

---

## 使用标准库的头文件

看下面这段程序：

```cpp
#include <iostream>

int main()
{
    std::cout << "Hello, world!";
    return 0;
}
```

程序通过 `std::cout` 输出 “Hello, world!”。但代码里**并没有给 `std::cout` 提供任何定义或声明**，编译器怎么知道 `std::cout` 是什么？

**答案是：`std::cout` 已经在头文件 `iostream` 中被前向声明了。** 当我们 `#include <iostream>` 时，就是在**请求预处理器**把这个叫 `iostream` 的文件里的所有内容（包括 `std::cout` 的前向声明）**复制到当前正在 `#include` 的文件里**。

> **关键洞察**
>
> `#include` 一个文件时，被包含文件的内容会**被插入到 `#include` 的位置**。这为我们提供了一种方便的方式，**把别的文件里的声明拉进来**。

想想如果 `iostream` 不存在会怎样：每次你用 `std::cout`，都得手动把与它相关的所有声明抄进当前文件顶部！这既需要你**了解 `std::cout` 是如何声明的**，又是**海量重复劳动**。更糟的是，一旦标准库里某个函数原型被新增或修改，你还得**手动更新**所有前向声明。

**所以直接 `#include <iostream>` 才是明智之选。**

---

## 用头文件“传播”前向声明

回到上一节的例子。上次我们有两个文件：

**`add.cpp`**：

```cpp
int add(int x, int y)
{
    return x + y;
}
```

**`main.cpp`**：

```cpp
#include <iostream>

int add(int x, int y); // 用函数原型做前向声明

int main()
{
    std::cout << "The sum of 3 and 4 is " << add(3, 4) << '\n';
    return 0;
}
```

（如果你在从零复现这个例子，别忘了把 `add.cpp` 加进项目里让它一起被编译。）

例子里我们用**前向声明**告诉 `main.cpp`：`add` 长这样。前面提过，**为每个跨文件调用的函数手写前向声明**很快就会变得繁琐不堪。

那我们干脆写一个**头文件**来省这份力气。写头文件其实很简单，它主要由两部分组成：

1. **头文件保护（header guard）**——下节详细讲；
2. **头文件的实际内容**——即你想让其他文件“看得见”的那些标识符的**前向声明**。

**新增头文件的操作**和新增源文件类似（见 [2.8](#)）。

- 使用 IDE：按新建源文件的步骤操作，但选择 **Header** 而不是 **Source**，新文件会作为项目的一部分出现。
- 使用命令行：在你喜欢的编辑器里、和源文件相同的目录中新建一个文件即可。**头文件不需要加进编译命令**，它们是通过 `#include` 隐式包含并随源文件一起编译的。

> **最佳实践**
>
> **头文件建议使用 `.h` 后缀**（除非你项目已经用了其它约定）。这是长期以来的 C++ 惯例，大多数 IDE 也默认使用 `.h`。

头文件常常**与源文件“成对出现”**：头文件提供对应源文件里定义的前向声明。由于我们的头文件要给 `add.cpp` 中定义的函数写前向声明，我们把它命名为 `add.h`。

> **最佳实践**
>
> **如果头文件与某个源文件“配对”**（例如 `add.h` 与 `add.cpp`），**二者应使用相同的基名**（`add`）。

一个完整的头文件示例：

**`add.h`**：

```cpp
// 这里其实应该写头文件保护，但为了简化先省略（下节详细讲）

// 这就是 .h 文件的内容——放声明的地方
int add(int x, int y); // add() 的函数原型 —— 别忘了分号！
```

要在 `main.cpp` 中使用这个头文件，我们需要 `#include` 它（**用双引号，不是尖括号**）。

**`main.cpp`**：

```cpp
#include "add.h"    // 把 add.h 的内容插入到这里（注意用双引号）
#include <iostream>

int main()
{
    std::cout << "The sum of 3 and 4 is " << add(3, 4) << '\n';
    return 0;
}
```

**`add.cpp`**：

```cpp
#include "add.h"    // 把 add.h 的内容插入到这里（注意用双引号）

int add(int x, int y)
{
    return x + y;
}
```

预处理器处理 `#include "add.h"` 时，会把 `add.h` 的内容复制到当前位置。因为 `add.h` 中包含了 `add()` 的前向声明，这条前向声明也就复制到了 `main.cpp` 中。最终效果与我们手动在 `main.cpp` 顶部写前向声明**完全等价**。

因此，程序能正确编译和链接。

> 提示：一些图示里“Standard Runtime Library”其实应标为 **“C++ Standard Library”**。

## 为什么“在头文件里写定义”会违反 ODR

**目前来说，不要把函数或变量的定义放到头文件里。** 一旦这个头文件被**多个源文件**包含，就会违反 **一次定义原则（ODR）**。

> **相关内容**
>
> ODR 参见 [2.7 — 前向声明与定义](#)。

来看会发生什么。

**`add.h`**：

```cpp
// 这里其实应该写头文件保护，为了简化先省略

// 在头文件里写函数定义 —— 不要这么干！
int add(int x, int y)
{
    return x + y;
}
```

**`main.cpp`**：

```cpp
#include "add.h"    // add.h 的内容会被复制到这里
#include <iostream>

int main()
{
    std::cout << "The sum of 3 and 4 is " << add(3, 4) << '\n';
    return 0;
}
```

**`add.cpp`**：

```cpp
#include "add.h"    // add.h 的内容会被复制到这里
```

编译 `main.cpp` 时，`#include "add.h"` 会被替换成 `add.h` 的内容，编译器实际编译的东西大致等价于：

```cpp
// 来自 add.h：
int add(int x, int y)
{
    return x + y;
}

// iostream 头文件的内容会在这里展开

int main()
{
    std::cout << "The sum of 3 and 4 is " << add(3, 4) << '\n';

    return 0;
}
```

这个 **`main.cpp` 单独编译** 没问题。

编译 `add.cpp` 时，`#include "add.h"` 也会被替换成：

```cpp
int add(int x, int y)
{
    return x + y;
}
```

这个 **`add.cpp` 单独编译** 也没问题。

但当**链接器**登场时，它会发现 `add()` 有**两个定义**：一个在 `main.cpp` 里，一个在 `add.cpp` 里。**这违反了 ODR 第 2 条**：“在同一程序内，一个变量或普通函数只能有一个定义。”

> **最佳实践**
>
> **目前不要把函数和变量的定义放到头文件里。** 一旦头文件被多个源文件 `#include`，就极易违反 ODR。

> **作者笔记**
>
> 后面我们会遇到一些**可以安全放在头文件中的定义**，例如**内联函数**、**内联变量**、**类型**、**模板**——它们对 ODR 有特殊豁免。等讲到时会详细说明。

## 源文件应该 `#include` 与它配对的头文件

C++ 的最佳实践是：**源文件应包含（`#include`）与它配对的头文件**（若存在）。这样一些错误就能被**编译期**捕获，而不是拖到**链接期**。例如：

**`add.h`**：

```cpp
// 这里其实应该写头文件保护，为了简化先省略

int add(int x, int y);
```

**`main.cpp`**：

```cpp
#include "add.h"
#include <iostream>

int main()
{
    std::cout << "The sum of 3 and 4 is " << add(3, 4) << '\n';
    return 0;
}
```

**`add.cpp`**：

```cpp
#include "add.h"         // 把 add.h 里的前向声明复制到这里

double add(int x, int y) // oops，返回类型误写成 double 了
{
    return x + y;
}
```

编译 `add.cpp` 时，`#include "add.h"` 会把 `int add(int x, int y)` 的前向声明复制进来。当编译器读到 `double add(int x, int y)` 这个**定义**时，会发现**返回类型与前向声明不一致**。由于函数不能仅凭返回类型来区分，编译器会**立即报错并终止**。在大项目里，这能省下大量排错时间。

> **顺便一提**
>
> 不巧的是，这一招**对“参数类型不同”不奏效**。因为 C++ 支持**函数重载**（同名不同参），编译器会把参数类型不同的当作**另一个重载**。没法两全。

如果没有 `#include "add.h"`，编译期就发现不了不一致，只能等到**链接期**才暴露。

后面还会看到很多例子——源文件所需的一些内容就定义在配对的头文件里，这种时候 `#include` 是**必须**的。

> **最佳实践**
>
> **源文件应 `#include` 与它配对的头文件**（若有）。

---

## 不要 `#include` `.cpp` 文件

预处理器不会拦你，但一般**不要 `#include` `.cpp` 文件**。`.cpp` 文件应作为项目的一部分被**独立编译**。原因包括：

- 可能引发源文件之间的**命名冲突**；
- 在大项目中容易踩 **ODR** 的坑；
- 一旦某个被 `#include` 的 `.cpp` 改动，**所有 `#include` 它的 `.cpp`** 都要重编，编译时间飙升。头文件通常比源文件改动得少；
- 不符合**惯例**。

> **最佳实践**
>
> **避免 `#include` `.cpp` 文件。**

> **小贴士**
>
> 如果你**不 `#include` `.cpp` 就编译不过**，说明那些 `.cpp` **根本没被加入项目**参与编译。把它们添加到项目/编译命令里让它们参与编译就好。

## 常见问题排查

- 编译器报 `add.h` **找不到** → 检查文件名是不是真的叫 `add.h`（有时会被存成 `add`（无后缀）、`add.h.txt`、`add.hpp` 等），并确保它和其它源文件在**同一目录**。
- 链接器报 **`add` 未定义** → 确保 `add.cpp` 已经**加入项目**，其中的 `add` 定义能被链接进来。

## 尖括号 vs 双引号

为什么 `iostream` 用 **尖括号 `<>`**，而 `add.h` 用 **双引号 `""`**？因为**可能存在同名头文件位于不同目录**。**尖括号和双引号**是给预处理器的“提示”，告诉它该到哪里去找。

- **尖括号 `<xxx>`**：表示“这是一个**不是我们自己写的**头文件”。预处理器**只会在“包含目录（include directories）”里查找**（包含目录通常是项目/IDE/编译器设置里的一项，默认包含随编译器、操作系统一同附带的头文件目录），**不会**在项目源码目录里找。
- **双引号 `"xxx"`**：表示“这是**我们自己写的**头文件”。预处理器会**先在当前目录**查找，找不到再去“包含目录”找。

> **规则**
>
> **自己写的头文件、或期望在当前目录中找到的头文件，用双引号。**
> **编译器/操作系统自带的头文件，以及安装在系统别处的第三方库头文件，用尖括号。**

---

## 为什么 `iostream` 没有 `.h` 后缀？

另一个常问的问题：为什么 `iostream`（以及许多标准库头文件）**没有 `.h` 后缀**？

答：`iostream.h` 与 `iostream` **是两个不同的头文件**。这要从一段小历史讲起。

早期 C++ 里，标准库所有头文件都以 `.h` 结尾，例如：

| 头文件类型 | 命名约定  | 举例         | 标识符所在命名空间 |
| ---------- | --------- | ------------ | ------------------ |
| C++ 专属   | `<xxx.h>` | `iostream.h` | 全局命名空间       |
| C 兼容     | `<xxx.h>` | `stddef.h`   | 全局命名空间       |

最早的 `cout`、`cin` 就是在 `iostream.h` 中、**位于全局命名空间**里声明的。世界很和谐。

后来 ANSI 委员会把 C++ 标准化时，决定**把标准库中所有名字都搬到 `std` 命名空间**，以减少和用户名字的冲突。但这样一来，那些原本 `#include <iostream.h>` 的老程序就都得改。

于是 C++ 引入了**不带 `.h`** 的新头文件，把所有名字**放在 `std` 命名空间**中。老程序继续用 `#include <iostream.h>`，新程序用 `#include <iostream>`。

现代 C++ 一共有 **4 组** 头文件：

| 头文件类型     | 命名约定  | 举例         | 标识符所在命名空间                           |
| -------------- | --------- | ------------ | -------------------------------------------- |
| C++ 专属（新） | `<xxx>`   | `iostream`   | `std` 命名空间                               |
| C 兼容（新）   | `<cxxx>`  | `cstddef`    | `std` 命名空间（必需）；全局命名空间（可选） |
| C++ 专属（旧） | `<xxx.h>` | `iostream.h` | 全局命名空间                                 |
| C 兼容（旧）   | `<xxx.h>` | `stddef.h`   | 全局命名空间（必需）；`std` 命名空间（可选） |

> **警告**
>
> 新的 C 兼容头文件 `<cxxx>` **可能**同时把名字放到全局命名空间；旧的 C 兼容头文件 `<xxx.h>` **可能**同时把名字放到 `std` 命名空间。**这些“可选位置”里的名字应尽量避免使用**，因为在不同实现里未必都存在。

> **最佳实践**
>
> **使用标准库头文件时不加 `.h` 后缀。** **用户自定义头文件仍然使用 `.h` 后缀。**

## 引用其它目录下的头文件

**（不推荐）** 一种做法是**在 `#include` 里直接写相对路径**：

```cpp
#include "headers/myHeader.h"
#include "../moreHeaders/myOtherHeader.h"
```

只要文件确实存在于对应位置就能编译，但**目录结构一变，代码全废**。

**（推荐）** 更好的做法是**告诉编译器/IDE：还有一些头文件在别处**，找不到时去那里找。这一般是通过 IDE 的项目设置**设置 include path / 搜索目录**来实现。

- **Visual Studio**：在 Solution Explorer 中右键项目 → **Properties → VC++ Directories → Include Directories**，添加要搜索的目录。
- **Code::Blocks**：**Project → Build Options → Search directories**，添加目录。
- **gcc**：命令行加 `-I`，例如：`g++ -o main -I./source/includes main.cpp`。`-I` 后面**不要有空格**；用绝对路径时把 `-I` 后的 `.` 去掉。
- **VS Code**：在 `tasks.json` 的 `Args` 里加一行，例如：`"-I./source/includes",`。规则同 gcc。

好处是：如果目录结构变了，**只需要改一个编译器/IDE 设置**，而不是每个源文件。

---

## 头文件里也可以 `#include` 其它头文件

头文件里的内容常会用到**另一个头文件**中声明或定义的东西。这时候，这个头文件**应该 `#include` 它所依赖的那个头文件**。

**`Foo.h`**：

```cpp
#include <string_view> // 使用 std::string_view 必需

std::string_view getApplicationName(); // 这里用到了 std::string_view
```

---

## “传递性包含（Transitive includes）”

当你的源文件 `#include` 了某个头文件时，那个头文件里 `#include` 的**其它头文件也会被间接引入**（继续递归下去）。这些被间接引入的头文件叫做 **传递性包含（transitive includes）**。

传递性包含带进来的内容**在你的源文件里“可用”**。但**一般不要依赖这些“间接带来的东西”**（除非文档明确说明这些传递性包含是必需的）。头文件的实现**可能会随版本或系统而变**。如果你依赖了它们，**代码可能只能在某些系统上编译，或者今天能编、明天不能编**。避免的办法很简单：**你用到什么，就显式 `#include` 什么**。

> **最佳实践**
>
> **每个文件都应显式 `#include` 它编译所需要的所有头文件。不要依赖其它头文件传递进来的包含。**

不幸的是，目前还没有简单的方法能自动检测“你是不是又不小心依赖了别的头文件传进来的东西”。

**Q：我并没有 `#include <someheader>`，我的程序也照样能跑，为什么？**

这是本站最常被问到的问题之一。答案通常是：**你 `#include` 了别的头文件（比如 `<iostream>`），那个头文件自己又 `#include` 了 `<someheader>`。** 虽然目前能编译，但按上面的最佳实践，**你不应该依赖这一点**——**在你机器上编得过，未必在朋友机器上也编得过**。

## `#include` 的顺序

如果每个头文件都写得规范（自己需要什么就 `#include` 什么），那么 `#include` 的顺序其实**无关紧要**。

但考虑一种情况：头文件 `A` 需要 `B` 中的声明，但**忘了 `#include B`**。在你的源文件里，如果 `#include B` 出现在 `#include A` **之前**，代码**仍然能编译**——因为编译器会先编译 `B` 里的声明，再编译 `A` 里依赖它们的内容。

但如果 `#include A` 出现在 `#include B` **之前**，编译器就会报错——因为轮到 `A` 时，它依赖的 `B` 中声明还没见过。**这其实是我们想要的**，因为**问题会被暴露出来**，好去修复它。

> **最佳实践**
>
> 为了让**缺失的 `#include`** 更容易被编译器发现，`#include` 的顺序建议如下（按需跳过不相关的项）：
>
> 1. **本源文件配对的头文件**（例如 `add.cpp` 先 `#include "add.h"`）
> 2. **本项目里的其它头文件**（例如 `#include "mymath.h"`）
> 3. **第三方库头文件**（例如 `#include <boost/tuple/tuple.hpp>`）
> 4. **标准库头文件**（例如 `#include <iostream>`）
>
> 每一组内部按**字母序**排序（除非某个第三方库文档另有要求）。
>
> 这样，如果你自己写的头文件缺少了对某个第三方库或标准库头文件的 `#include`，就更容易触发编译错误，让你尽早修复。

## 头文件的最佳实践

再总结一些编写和使用头文件的建议：

- **始终写头文件保护（header guard）**（下节讲）。
- **目前不要在头文件里定义变量或函数**。
- **头文件与其配对的源文件同名**（例如 `grades.h` 配 `grades.cpp`）。
- **一个头文件只负责一件事**，尽量独立。比如把功能 A 相关的声明放 `A.h`，功能 B 相关的放 `B.h`。以后只需要用 A 时，只 `#include "A.h"`，不用连带把 B 的东西也拉进来。
- **注意显式 `#include` 你实际用到的头文件**，避免不自觉地依赖传递性包含。
- **一个头文件应 `#include` 它自己所需要的其它头文件**——它单独被 `#include` 进一个 `.cpp` 时也应能顺利编译。
- **只 `#include` 你真正需要的**，不要图省事什么都塞进来。
- **不要 `#include` `.cpp` 文件**。
- **把“做什么、怎么用”的文档放在头文件里**（读者更容易看到）；**“内部如何实现”的说明放在源文件里**。
