# 1.6 — 未初始化的变量与未定义行为

---

## 未初始化的变量

和某些编程语言不同，**C/C++ 并不会自动把大多数变量初始化为某个给定的值**（例如 0）。当一个**没有被初始化**的变量被分配到一块内存地址时，这个变量的“默认值”**就是那块内存里恰好残留着的（垃圾）值**！**没有通过初始化或赋值获得一个已知值的变量**，被称为 **未初始化变量（uninitialized variable）**。

> **命名约定**
>
> 许多读者以为“已初始化”和“未初始化”是严格相反的，但**它们并不完全对称**！在通常的用法里：
>
> - **已初始化（initialized）**：对象**在定义时**被给了一个初始值；
> - **未初始化（uninitialized）**：对象**还没有通过任何方式**（包括赋值）获得一个已知值。
>
> 因此，一个**没有被初始化、但之后被赋值**的对象**就不再是未初始化的**了（因为它已经有了一个已知值）。
>
> 小结：
>
> - **Initialized（已初始化）** = 对象在**定义时**被赋予了一个已知值；
> - **Assignment（赋值）** = 对象在**定义之后**被赋予了一个已知值；
> - **Uninitialized（未初始化）** = 对象**至今**还没被赋予任何已知值。

相关地，考虑这条变量定义：

```cpp
int x;
```

在 [1.4 — 变量的赋值与初始化](1.4_变量的赋值 和初始化.md) 中我们提到过：如果没有提供初始化器，那么变量会经历**默认初始化（default-initialization）**。在大多数情况下（比如这个例子），**默认初始化实际上什么初始化都不会做**。因此我们会说 `x` 是**未初始化的**。我们关注的是**结果**（对象没有已知的值），而不是**过程**。

> **顺便一提**
>
> 这种“不默认初始化”的行为，其实是 C 时代遗留下来的一种**性能优化**——当时的计算机很慢。想象你要从文件读入 100000 个值：你会先创建 100000 个变量，再把文件中的数据依次填进去。
>
> 如果 C++ 在创建时把这 100000 个变量都做默认初始化，就会**多做 100000 次初始化**（很慢），但收益很小（反正马上就被覆盖掉了）。
>
> 就现在而言，你都应该**始终初始化变量**，因为它的代价相比收益来说微不足道。等你更熟练之后，某些场景下你可能会为了性能而故意不做初始化，**但这必须是有选择、有意识地进行**。

**使用未初始化变量的值会导致意料之外的结果。** 看下面这段简短程序：

```cpp
#include <iostream>

int main()
{
    // 定义一个名为 x 的整型变量
    int x; // 这个变量是未初始化的，我们没有给它任何值

    // 把 x 的值打印出来
    std::cout << x << '\n'; // 天知道会输出什么，因为 x 是未初始化的

    return 0;
}
```

这里，计算机会给 `x` 分配一块未使用过的内存，然后把那块内存里的值（按整数解释）送给 `std::cout`。那么会打印出什么呢？**答案是“谁也不知道！”**，而且每次运行的结果**可能会（也可能不会）不同**。我在`vscode`上运行结果是0。你完全可以自己编译运行（电脑不会爆炸）。

如果你的编译器**根本不让**你编译并运行上面这段程序（例如它把这种情况直接当作错误），下面这种写法可以“绕过”这个检查：

```cpp
#include <iostream>

void doNothing(int&) // 现在不用管 & 是什么，我们只是用它骗编译器相信 x 被使用过
{
}

int main()
{
    // 定义一个名为 x 的整型变量
    int x; // 这个变量是未初始化的

    doNothing(x); // 让编译器以为我们给这个变量赋过值

    // 打印 x 的值（谁也不知道会是什么，因为 x 是未初始化的）
    std::cout << x << '\n';

    return 0;
}
```

**使用未初始化的变量是新手最常犯的错误之一**，而且不幸的是，它也可能是最难调试的错误之一——因为如果那块内存里恰好残留了一个“看起来合理”的值（比如 0），程序**可能照样能正常运行**。

这正是我们强调 **“永远初始化你的变量”** 这条最佳实践的主要原因。

## 未定义行为（Undefined Behavior）

“使用未初始化变量的值”是我们遇到的**第一个未定义行为的例子**。**未定义行为（undefined behavior，通常缩写为 UB）** 指的是：**执行了一段 C++ 语言并未明确规定其行为的代码所产生的结果**。在这个例子中，C++ 没有任何规则规定“**当你使用一个尚无已知值的变量时会发生什么**”。因此，一旦你这么做了，就会触发未定义行为。

包含未定义行为的代码可能出现下列任意一种“症状”：

- 每次运行结果都不一样；
- 每次都稳定地输出**同一个错误的结果**；
- 表现**时对时错**（有时正确，有时错误）；
- 看起来**一切正常**，但**在程序后面某处**产生了错误结果；
- 程序崩溃，可能立刻崩，也可能晚一点崩；
- 在某些编译器上工作，另一些上就不工作；
- 一直好好的，直到你改了**看似无关**的另一段代码，它就出问题了；
- ……或者，**它就是能给你正确的结果**（真的碰巧对了）。

> **作者笔记**
>
> **未定义行为就像一盒巧克力：你永远不知道下一颗是什么味的！**

C++ 中有很多情况**一不小心就会**导致未定义行为。我们会在后续课程中每次遇到时给你标出来。**记住这些坑，并小心避开它们。**

> **规则**
>
> **务必小心避免所有会导致未定义行为的情况**，比如使用未初始化变量。

## 实现定义行为（Implementation-defined）与未指定行为（Unspecified）

一个具体的**编译器**及其配套的**标准库**合起来称为一个**实现（implementation）**（因为它们才是真正在“实现” C++ 这门语言的东西）。有些情况下，C++ 标准**允许由实现来自行决定**某些方面的行为，好让编译器能选择在特定平台上更高效的做法。这种由实现决定的行为叫做**实现定义行为（implementation-defined behavior）**。**同一个实现内，实现定义行为必须是文档化的、且保持一致的。**

来看一个简单的例子：

```cpp
#include <iostream>

int main()
{
    std::cout << sizeof(int) << '\n'; // 打印 int 类型占多少字节

    return 0;
}
```

在大多数平台上会输出 `4`，但在某些平台上可能是 `2`。

> **相关内容**
>
> 关于 `sizeof()`，我们在 4.3 — 对象的大小与 sizeof 运算符 中讨论。

**未指定行为（unspecified behavior）** 和实现定义行为几乎一样，都是把具体行为**交给实现去决定**，但**不要求实现给出文档说明**。

我们通常应该**尽量避免**实现定义行为和未指定行为，因为这意味着你的程序在**换一个编译器**（甚至**只是改动了一些会影响实现行为的项目设置**）后，可能就无法按预期工作了。

> **最佳实践**
>
> **尽可能避免实现定义行为和未指定行为**，因为它们可能导致你的程序在其它实现上无法正确运行。
