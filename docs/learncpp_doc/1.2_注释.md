# 1.2 — 注释（Comments）

---

**注释（comment）** 是一段直接插入到程序源代码中、供程序员阅读的说明性文字。注释会被编译器**忽略**，仅供程序员自己使用。

在 C++ 中有两种不同风格的注释，但它们的作用是一样的：**帮助程序员在某种程度上说明代码**。

---

## 单行注释（Single-line comments）

`//` 符号表示一段 C++ **单行注释**，它告诉编译器忽略从 `//` 开始到本行末尾的所有内容。例如：

```cpp
std::cout << "Hello world!"; // 从这里到行尾的内容都会被忽略
```

通常，单行注释用于对某一行代码做简短的说明。

```cpp
std::cout << "Hello world!\n"; // std::cout 位于 iostream 库中
std::cout << "It is very nice to meet you!\n"; // 这些注释让代码变得很难读
std::cout << "Yeah!\n"; // 尤其是当每行长度不一致时
```

把注释放在代码右边可能会让代码和注释都变得难以阅读，特别是在行比较长时。如果每行都比较短，可以把注释**对齐**（通常对齐到某个制表位），像这样：

```cpp
std::cout << "Hello world!\n";                 // std::cout 位于 iostream 库中
std::cout << "It is very nice to meet you!\n"; // 这样读起来舒服多了
std::cout << "Yeah!\n";                        // 你说是不是？
```

但如果代码行较长，右侧再加注释就会让整行变得非常长。这种情况下，单行注释通常会写在**被注释代码的上方**：

```cpp
// std::cout 位于 iostream 库中
std::cout << "Hello world!\n";

// 这样读起来舒服多了
std::cout << "It is very nice to meet you!\n";

// 你说是不是？
std::cout << "Yeah!\n";
```

## 多行注释（Multi-line comments）

`/*` 和 `*/` 这一对符号表示 **C 风格的多行注释**。它们之间的所有内容都会被忽略。

```cpp
/* 这是一段多行注释。
   这一行会被忽略。
   这一行也会被忽略。 */
```

由于两个符号之间的所有内容都会被忽略，你有时会看到程序员把多行注释“美化”成这样：

```cpp
/* 这是一段多行注释。
 * 左边对齐的星号
 * 可以让它读起来更清晰
 */
```

**多行注释不能嵌套。** 因此下面这段代码会产生意料之外的结果：

```cpp
/* This is a multi-line /* comment */ this is not inside the comment */
// 上面这段注释在“第一个 */”处就结束了，而不是“第二个 */”
```

编译器编译这段代码时，会忽略从第一个 `/*` 到第一个 `*/` 之间的内容。由于 `this is not inside the comment */` **不算在注释里**，编译器会尝试编译它，结果必然会出现编译错误。

这也是**语法高亮**很有用的地方之一——注释会用不同的颜色显示，一眼就能看出哪些是注释、哪些不是。

> **警告**
>
> 不要在多行注释里再嵌套多行注释。但**在多行注释里包含单行注释是可以的**。

---

## 正确地使用注释

通常，注释主要用于以下三个用途。

### 1. 说明「做什么」

对于一个库、程序或函数来说，注释最适合用来说明**这个库、程序或函数是做什么的**。它们通常放在文件、库的顶部，或函数的正上方。例如：

```cpp
// 本程序根据学生的考试成绩和作业成绩来计算最终成绩。
// 这个函数使用牛顿法来近似求解给定方程的根。
// 以下几行根据稀有度、等级和权重因子生成一个随机物品。
```

这些注释让读者不用看具体代码，就能大致了解这个库、程序或函数想做什么。使用者（可能是别人，也可能是想复用旧代码的你自己）能一眼看出这段代码是否与自己要做的事情相关。这一点在**团队合作**中尤为重要，因为不是每个人都熟悉所有代码。

### 2. 说明「怎么做」

在库、程序或函数内部，可以使用注释来描述代码**打算如何**实现它的目标：

```cpp
/* 计算最终成绩时，我们会把所有加权后的期中成绩和作业成绩加起来，
   然后除以成绩数量得到一个百分比，再用这个百分比换算成字母等级。 */
```

```cpp
// 生成随机物品的做法如下：
// 1) 把所有指定稀有度的物品放进一个列表
// 2) 根据等级和权重因子计算每个物品的概率
// 3) 生成一个随机数
// 4) 找出这个随机数对应哪个物品
// 5) 返回相应的物品
```

这些注释让读者能大致理解代码**是如何**达成目标的，而不需要读懂每一行代码。

### 3. 说明「为什么」

在**语句层面**，注释应该用来说明代码**为什么**这样写。一个不好的语句级注释是**只说明代码在做什么**。如果你写的代码复杂到必须用注释来解释它在做什么，那你大概应该**重写这条语句**，而不是给它加注释。

下面是一些「不好的行级注释」和「好的语句级注释」的例子。

**不好的注释：**

```cpp
// 把 sight 设为 0
sight = 0;
```

原因：我们一眼就能看出来 `sight` 被赋成 0 了。

**好的注释：**

```cpp
// 玩家刚刚喝下了失明药水，什么都看不见了
sight = 0;
```

原因：现在我们知道**为什么** `sight` 被设为 0。

**不好的注释：**

```cpp
// 计算物品总价
cost = quantity * 2 * storePrice;
```

原因：我们能看出这是在算价格，但为什么 `quantity` 要乘以 2？

**好的注释：**

```cpp
// 这里要把数量乘以 2，因为它们是成对出售的
cost = quantity * 2 * storePrice;
```

原因：现在我们知道这个公式为什么合理。

程序员经常要在“用这种方式解决问题”和“用另一种方式解决问题”之间做出艰难的选择。注释是一种很好的方式，用来**提醒自己（或告诉他人）为什么选择了 A 方案而不是 B 方案**。

**好的注释：**

```cpp
// 我们决定用链表而不是数组，
// 因为数组的插入操作太慢。
```

```cpp
// 我们打算用牛顿法来求这个数的根，
// 因为这类方程没有确定性的解析解法。
```

最后，注释应当写成**即使一个完全不了解代码的人也能看懂**的样子。程序员经常会说：“这段代码显而易见嘛！我怎么可能忘！”。结果呢？它并不显然，而且你会惊讶地发现自己忘得有多快。:) 用自然语言把代码的**做什么、怎么做、为什么**写下来，未来的你（或别人）会感激你。**阅读单行代码很容易，但理解它们要达成什么目标却不容易。**

>
> 关于变量声明语句的注释，我们会在 1.7 —— 关键字与标识符命名中讨论。

> **最佳实践**
>
> **多写注释**，并且写注释时把读者当作一个完全不了解这段代码的人。**不要假设你自己会记得当初为什么这么做。**

## 注释掉代码（Commenting out code）

把一行或多行代码变成注释，叫做**注释掉代码（commenting out）**。这提供了一种便捷的方式，可以**临时**把某部分代码排除在编译之外。

要注释掉单行代码，直接用 `//` 风格的注释就行：

未注释掉：

```cpp
std::cout << 1;
```

已注释掉：

```cpp
//    std::cout << 1;
```

要注释掉一段代码，可以在多行前面都加上 `//`，也可以用 `/* */` 把它们整个包住。

未注释掉：

```cpp
std::cout << 1;
std::cout << 2;
std::cout << 3;
```

已注释掉（方式一）：

```cpp
//    std::cout << 1;
//    std::cout << 2;
//    std::cout << 3;
```

已注释掉（方式二）：

```cpp
/*
    std::cout << 1;
    std::cout << 2;
    std::cout << 3;
*/
```

这么做的原因有很多：

- **你正在写一段还无法编译的新代码，但又需要先运行程序。** 编译器不允许存在编译错误的代码通过编译，把那段无法编译的代码注释掉，就能让程序编译成功，方便你运行。等准备好了，再把注释去掉继续开发。
- **你写的新代码能编译，但工作不正常，暂时又没时间修。** 把有问题的代码注释掉，可以保证它不会执行、不会引发其它问题，等你有时间再修。
- **用来定位错误来源。** 如果程序结果不对（或崩溃），有时可以通过**禁用一部分代码**来判断问题出在哪。如果注释掉几行后程序恢复正常（或不再崩溃），那问题很可能就在你刚注释掉的那几行里，然后你就可以针对性地排查。
- **你想用一段代码替换另一段代码。** 与其直接删除原代码，不如先注释掉，作为参考保留在那里，等新代码确认工作正常后再删掉。如果新代码不行，还可以随时把它删掉，把老代码恢复回来。

因为在开发中经常需要“注释掉代码”，很多 IDE 都提供了对选中代码进行注释/取消注释的快捷功能。具体入口因 IDE 而异。

> **VS Code 用户**
>
> 按 **Ctrl + /** 即可注释/取消注释选中的代码。
